# 第一部分：系统概述与核心规则

## 1\. 系统概述



本系统服务于 BNBU（北京师范大学\-香港浸会大学联合国际学院）的体育 PE 课程管理。核心业务是**学生运动打卡**和**期末成绩评定**。



### 1\.1 核心业务规则（全局）



|规则项|值|说明|
|---|---|---|
|总打卡时长|20h|本学期需完成的打卡总时长|
|课程运动最低|10h|可被校队/社团抵扣（与自主其他运动合计最多抵扣 20h）|
|自主其他运动最低|10h|可被校队/社团抵扣（与课程运动合计最多抵扣 20h）|
|每日提交上限|2h|每日最多申报的打卡时长|
|每日提交次数|1 次|每日只能提交一次打卡|
|学时步进|仅 1h / 2h|每次打卡只能选 1 小时或 2 小时|



### 1\.2 三端覆盖



|端|用户|形态|
|---|---|---|
|**Web 前端**|学生 / 授课老师 / 管理员|浏览器 SPA|
|**Android**|学生|原生 App \(Kotlin/Compose\)|
|**iOS**|学生|原生 App \(SwiftUI\)|



### 1\.3 重要术语



|术语|说明|
|---|---|
|**打卡时长**|学生提交的运动时长（1h 或 2h）|
|**期末审查**|教师不再逐条审核打卡，期末集中审查，学期中也可以查看|
|**课程任务**|已删除，不再有任务发布功能|
|**课程运动**|在体育课程下提交的打卡|
|**自主其他运动**|自由运动打卡，可用校队/社团认证抵扣（课程运动也可抵扣）|



### 1\.4 数据展示原则



> **所有前端展示的数据必须来源于后端返回的真实数据**，不硬编码、不伪造、不假设默认值。
> 
> 



- 学生打卡时长: 来源于数据库中实际存储的学生提交记录

- 成绩项目与权重: 来源于教师发布的最新版本成绩规则

- 抵扣后剩余时长: 来源于教师确认抵扣后的实际计算结果

- 耐力跑换算分数: 来源于管理员配置的换算表 \+ 学生实际用时

- 学生性别/年龄/年级: 来源于数据库中学生档案信息

- 未录入的成绩项目: 展示"未录入"，不展示 0 或占位数字

    

### 1\.5 隐私合规



- **隐私政策同意**: 学生首次登录时弹窗展示《隐私政策》，必须点击同意才能进入系统。系统记录同意时间和政策版本号

- **GPS 数据保留**: 运动打卡附带的位置信息仅保留 **90 天**，到期自动清除。教师审查时在 90 天内可查看位置，过后仅显示"位置信息已过期"

- **推送通知**: 推送通知不包含个人信息（如姓名、成绩分数），仅发通用通知（如"你的成绩已发布，请登录查看"）

- **数据删除**: 学生可在「个人设置」中提交**数据删除申请**，管理员审批后执行。采用**分层删除**策略：

    - **删除范围**：个人打卡记录、凭证文件等非核心个人数据

    - **保留范围**：完成成绩评定、申诉处理和档案归档义务所需的最小数据（成绩结果、申诉记录、归档快照），保留至毕业后 5 年

    - 已归档课程中的成绩数据不可删除（已锁定为学校档案）

    - **匿名化处理**：保留的学时统计数据做匿名化处理（不影响课程整体数据），删除后学生身份与保留数据不可关联

- **数据最小化**: 系统仅收集完成体育打卡和成绩评定所必需的数据，不收集不必要的个人信息

    

### 1\.6 多语言支持



学生端 Web / App 支持**中文（简体）和English**双语切换：



- **界面语言**: 导航、按钮、表单标签、提示信息等全部 UI 文本支持中英双语

- **语言检测**: 首次启动根据系统语言自动选择，可在「个人设置」中手动切换

- **邮件模板**: 系统邮件（密码设置、忘记密码、打卡截止提醒）根据学生语言偏好自动选择中/英文版本

- **教师端和管理端**: 现阶段仅支持中文

- **数据内容**: 成绩项目名称、课程名称等教师/管理员录入的内容不做自动翻译

    

### 1\.7 国际化学生支持



- **英文界面**: 留学生默认使用英文界面（根据系统语言自动检测），也可在设置中手动切换

- **英文邮件**: 系统邮件模板已支持中英双语，根据学生语言偏好自动选择对应语言版本

- **体测标准**: 留学生同样适用中国国家体测标准——在校学习期间统一要求，不做差异化处理

- **英文支持范围**: 学生端全部 UI（导航、表单、提示、错误信息等）均支持英文；教师端和管理端现阶段仅支持中文

    

## 14\. 打卡时长规则配置



### 14\.1 业务概述



管理员可配置全局体育规则。



|规则项|默认值|说明|
|---|---|---|
|总打卡时长|20h|本学期需完成的总打卡时长|
|课程运动最低|10h|可被校队/社团抵扣（与自主其他运动合计最多抵扣 20h）|
|自主其他运动最低|10h|可被校队/社团抵扣（与课程运动合计最多抵扣 20h）|
|每日提交上限|2h|每日最多提交时长|
|每日提交次数|1 次|每日只能提交一次|



### 14\.2 规则生效机制



```Plain Text
管理员修改规则 → 选择生效范围：
  ├── 「仅新学期生效」（推荐）：当前学期不变，下学期自动应用新规则
  └── 「立即生效」（高风险）：当前学期所有学生立即按新规则计算
```



### 14\.3 规则版本记录



每次修改记录：版本号、修改人、修改时间、变更内容（旧值→新值），可追溯。



### 14\.4 课程级覆盖



暂不支持按课程覆盖全局规则。所有课程统一适用全局规则。如未来有特殊体育课（如康复体育课）需要不同规则，再考虑加入课程级规则覆盖。



## 15\. 打卡时间窗



### 15\.1 业务概述



管理员可设置全局默认打卡时间窗，任课教师可为自己的课程单独设置时间窗。课程级时间窗优先于全局时间窗。学生只能在所属课程对应的时间窗内打卡。



### 15\.2 时间窗配置



|配置项|说明|示例|
|---|---|---|
|窗口模式|「学期全程开放」或「指定时间段」|2026\.02\.24 \~ 2026\.06\.28|
|每日时段|每天允许打卡的时间段|06:00 \~ 22:00|
|特殊日排除|排除特定日期（考试周等）|2026\.04\.20 \~ 2026\.04\.26|



### 15\.3 时间窗层级



- **全局默认时间窗**：管理员设置，适用于未单独配置的课程

- **课程级时间窗**：任课教师可为自己的课程单独设置时间窗，覆盖全局默认

- 课程级时间窗优先于全局默认时间窗

- 暂不支持多个不连续的每日时段（如早\+晚两个时段）

    

### 15\.4 节假日处理



节假日不自动排除——如果管理员需要关闭打卡，手动修改时间窗配置。每次修改记录历史，可回滚。



### 15\.5 设置历史



- 每次修改时间窗记录历史

- 可查看历史配置记录

### 2\.3 教师账号生命周期管理



#### 创建流程



```Plain Text
管理员登录 → 「用户管理」→ 「添加教师」
  → 输入：工号（必填）、姓名（必填）、邮箱（必填）
  → 系统自动发送密码设置邮件（与学生流程相同）
  → 教师点击邮件链接 → 设置密码 → 登录
```



**创建规则**:



- 教师账号由管理员在「用户管理」中统一创建，教师无法自行注册

- 工号为唯一标识，不可重复

- 邮箱用于接收密码设置邮件和系统通知，默认使用学校邮箱域名

- 密码设置流程与学生完全一致：邮件中的链接 7 天有效，过期后可通过「忘记密码」重新获取

    

#### 账号状态



账号仅有两种核心状态：**激活**和**未激活**。不存在"冻结"状态。



|状态|含义|触发条件|
|---|---|---|
|**激活**|可正常使用|密码已设置，账号可正常登录使用|
|**未激活**|未设置密码|管理员创建后、用户尚未点击邮件链接设置密码|



**状态流转**:



```Plain Text
管理员创建账号 → 未激活
  → 用户点击邮件设置密码 → 激活
    ├── 连续 5 次密码错误 → 临时锁定（15 分钟后自动恢复，不改变激活状态）
    └── 管理员手动禁用 → 管理员可重新启用（不改变激活状态）
```



> **说明**：连续密码错误导致的临时锁定是安全机制，不改变账号的激活/未激活状态。管理员对账号的禁用/启用操作也不改变核心状态。
> 
> 



#### 密码重置



教师端登录页同样有「忘记密码」入口，流程与学生完全一致：



```Plain Text
登录页 → 点击「忘记密码」→ 输入工号
  → 系统发送重置邮件到教师邮箱（30 分钟有效）
  → 教师点击链接 → 设置新密码 → 登录
```



### 2\.4 管理员账号管理



#### 设计原则：约定优于配置（Convention over Configuration）



管理员权限采用**预设角色 \+ 可配置权限边界**的方案：



- 系统预设两个管理员角色（超级管理员、体育部管理员），各自有合理的**默认权限集**

- 超级管理员可以在界面上调整体育部管理员的权限范围——开启或关闭特定权限项

- 权限判断遵循"有自定义配置按配置，无配置使用默认值"的规则

    

**选择此方案的理由**：



- 比完全硬编码（两个角色权限写死在代码里）更灵活，适应运维中的实际需求变化

- 比完整 RBAC（自由创建角色 \+ 权限点组合）更简单，避免为 4 个角色引入企业级权限系统的复杂度

- 默认配置即可覆盖 80% 场景，剩下的 20% 通过界面调整即可满足，无需改代码或发版

    

#### 角色定义



|管理员类型|默认权限范围|说明|
|---|---|---|
|**超级管理员**|全部功能|系统初始化时内置，通常为 IT 部门人员。不可删除、不可降级|
|**体育部管理员**|业务管理|学期/课程/成绩归档管理，**默认不能**管理用户账号和系统配置|



#### 默认权限边界（可通过界面调整）



下表为体育部管理员的**默认**权限。超级管理员在「系统配置 → 管理员权限」界面中可逐项开启或关闭：



|操作|超级管理员|体育部管理员（默认）|体育部管理员（可开启）|
|---|---|---|---|
|学期管理|✅|✅|—|
|课程管理|✅|✅|—|
|成绩归档审核|✅|✅|—|
|学时规则配置|✅|✅|—|
|耐力跑换算表管理|✅|✅|—|
|系统公告发布|✅|✅|—|
|审计日志查看（仅业务）|✅|✅|—|
|用户管理（创建/停用/解锁账号）|✅|❌|✅|
|系统配置（SMTP/COS/学校信息）|✅|❌|✅|
|创建/删除管理员|✅|❌|❌|
|审计日志查看（全部）|✅|❌|✅|



> **关于"创建/删除管理员"**：此项为超级管理员**独占**，体育部管理员在任何情况下都无法获得。所有管理员变更操作均记录审计日志。
> 
> 



#### 创建规则



- **超级管理员**: 系统初始化时在数据库层面创建（不可通过界面删除）。通过环境变量指定初始密码，首次登录后强制修改

- **体育部管理员**: 由超级管理员在「用户管理」中创建，流程同教师账号创建。创建后默认拥有"默认"列的全部权限

- **管理员数量**: 超级管理员 ≥ 2 人（避免单人故障导致系统无法管理）；体育部管理员不做硬性限制

- **安全约束**:

    - 超级管理员不可删除自己

    - 超级管理员不可将自己的角色降级为体育部管理员

    - 体育部管理员无法访问「管理员权限配置」界面

    - "创建/删除管理员"为超级管理员独占，体育部管理员在任何情况下都无法获得此项权限

    - 所有管理员操作（含权限配置变更）均记录审计日志

        

#### 典型运维场景



|场景|操作|
|---|---|
|开学季，体育部需要批量创建 50 个教师账号|超级管理员临时为体育部管理员开启"用户管理"权限，操作完成后关闭|
|体育部管理员需要排查邮件发送问题|超级管理员临时开启"系统配置"可见权限，排查完后关闭|
|体育部管理员离职，新任接手|删除旧账号，创建新账号（权限用默认配置即可，无需额外调整）|
|学校引入第三方审计|超级管理员为体育部管理员开启"审计日志（全部）"查看权限，审计期结束后关闭|

## 3\. 认证与授权体系



### 3\.1 登录流程（三端统一）



```Plain Text
学生/教师/管理员 → 输入账号（学号/工号）+ 密码
  → 后端校验 → 签发 JWT（含 userId、role、exp）
    → 前端存储 Token
      → 后续请求携带 Authorization: Bearer <token>
```



### 3\.2 Token 生命周期



|参数|值|说明|
|---|---|---|
|Access Token 有效期|24h|短期，降低泄露风险|
|Refresh Token|不需要|当前架构无 refresh token，过期重新登录|
|Token 存储位置|Web: localStorage；App: EncryptedSharedPreferences / Keychain|App 端使用系统安全存储|



### 3\.3 多设备登录



**允许同一账号同时在 Web \+ Android \+ iOS 登录，不互踢。** 学生可能在电脑上看成绩、手机上打卡，踢掉任何一个都不合理。Token 各自独立签发，互不影响。



### 3\.4 账号锁定



|规则|值|
|---|---|
|连续失败次数|5 次|
|锁定时间|15 分钟|
|锁定范围|按账号（非 IP）|
|解锁方式|自动过期 或 管理员手动解锁|



### 3\.5 密码修改（已登录状态）



```Plain Text
学生端「个人设置」→ 「修改密码」
  → 输入旧密码 + 新密码 + 确认新密码
    → 后端校验旧密码正确 → 更新密码 → 不强制重新登录
```



### 3\.6 登出行为



**仅清本地 Token**，服务端不维护会话状态（无状态 JWT）。不需要服务端感知登出。



### 3\.7 密码设置（新生 \+ 忘记密码）



详见第七部分「教师端课程管理：名单导入与账户体系」中的密码与邮件流程。

# 第三部分：学生打卡业务



> **来源**: 模块3\-学生打卡业务\_刘一宽\_逐项审核报告\(1\)\.md
> 
> 



## 1\. 业务目标



学生通过开始运动、结束运动和提交凭证，形成可计入本学期体育学时的运动记录。学生可随时查看已提交记录和累计学时，任课教师在学期打卡截止后集中审查记录并修正最终有效学时。



学生打卡分为两个学时类别：



- **课程相关运动**：计入课程运动学时，学期要求为 10 小时，不可由校队或社团认证抵扣。

- **其他运动**：计入自主其他运动学时，学期要求为 10 小时，可由已确认的校队或社团认证抵扣。

    

两个类别合计构成本学期 20 小时体育学时要求。



## 2\. 参与角色



### 2\.1 学生



- 查看当前学期的打卡开放状态、已完成学时和剩余学时。

- 选择打卡类别和运动项目。

- 开始并结束运动计时。

- 填写运动说明并提交图片或视频凭证。

- 查看本人打卡记录、学时变化和教师公开反馈。

- 对教师期末处理结果存在异议时，通过课程既有沟通渠道联系任课教师复核。

    

### 2\.2 任课教师



- 负责本人课程学生的课程相关运动和其他运动记录。

- 在学期打卡截止后集中查看学生记录和运动凭证。

- 将不真实、不合规或无法证明的记录标记为无效。

- 恢复被误标的记录，或按实际运动情况修正记录学时。

- 添加仅教师可见的内部备注，或向学生公开反馈。

- 对大量异常记录进行集中处理，必要时联系体育部管理员。

    

### 2\.3 体育部管理员



- 设置全校统一的学期学时规则和打卡时间范围。

- 设置每天允许开始打卡的时段和停止打卡的特殊日期。

- 处理严重或批量违规情况，并可限制相关账号继续打卡。

- 维护全校统一的运动项目范围和数据管理规则。

    

## 3\. 核心业务规则



### 3\.1 学时规则



- 每学期总要求为 20 小时，其中课程相关运动 10 小时、其他运动 10 小时。

- 校队/社团认证可抵扣课程运动学时和/或自主其他运动学时，由教师自定义分配，合计最多抵扣 20 小时。

- 每个自然日最多成功提交 1 次打卡。

- 每个自然日最多计入 2 小时。

- 学生当天提交 1 小时后，当天仍视为已完成一次打卡，不能再次提交。

- 打卡日期以开始运动时所在的自然日为准。

- 打卡跨越零点时，记录仍归属于开始运动的日期，不占用结束运动日期的次数。

    

### 3\.2 计时与学时换算



- 学生点击"开始运动"后开始计时。

- 运动不足 1 小时，不计入学时，也不能形成成功打卡记录。

- 运动满 1 小时但不足 2 小时，计入 1 小时。

- 运动满 2 小时，计入 2 小时并自动结束本次计时。

- 自动结束时应明确提醒学生计时已达到当日上限，并引导学生补充说明和凭证。

- 学时由实际开始时间和结束时间换算，学生不再手动选择 1 小时或 2 小时。

    

### 3\.3 打卡时间范围



- 学生只能在当前学期已开放的日期和每日时段内开始运动。

- 被排除的日期不能开始新的运动计时。

- 学生在开放时段内已经开始运动后，即使结束时间超过每日开放时段，也可正常结束和提交。

- 学期打卡截止后不能开始或补交新的打卡记录。

    

### 3\.4 多端操作



- 同一学生同一时间只能存在一条进行中的运动计时。

- 学生在一端开始运动后，其他端再次开始时显示"已有进行中的运动记录"。

- 多端同时提交时，以最先成功提交的记录为准，其余提交显示"今日已打卡"。

- 草稿只保存在创建草稿的设备，不在不同设备间同步。

    

## 4\. 打卡前置条件



学生进入打卡页面时依次完成以下检查：



1. 学生账号处于可正常使用状态，且未被限制打卡。

2. 当前日期位于本学期允许打卡的日期范围内。

3. 当前时间位于每日允许开始打卡的时段内。

4. 学生在开始运动日期尚无成功打卡记录。

5. 学生当前没有在其他设备上进行中的运动计时。

6. 课程相关运动要求学生已选中当前学期唯一一门体育课程。

    

任一条件不满足时，页面直接说明原因，不进入计时流程。



## 5\. 学生正常打卡流程



### 5\.1 查看当日状态



学生进入打卡页面后看到：



- 当前学期课程名称和任课教师。

- 课程相关运动、其他运动和总学时的完成进度。

- 今日是否已经成功打卡。

- 当前是否允许开始运动。

- 如存在进行中的运动，显示开始时间、已运动时长和继续进入按钮。

    

### 5\.2 选择打卡类别



学生在开始运动前选择以下一种类别：



#### 课程相关运动



- 自动关联学生当前学期唯一一门体育课程。

- 适用于该体育课程要求、布置或认可的课外运动。

- 不再选择课程任务。

- 记录计入课程相关运动学时。

    

#### 其他运动



- 不关联具体课程任务。

- 由学生当前体育课程的任课教师统一负责期末审查。

- 记录计入其他运动学时。

- 已确认的校队或社团抵扣只作用于其他运动学时要求，不生成学生打卡记录，也不改变课程相关运动学时。

### 5\.3 选择运动项目



学生选择以下一种运动项目：



- 跑步

- 篮球

- 足球

- 羽毛球

- 游泳

- 健身

- 骑行

- 其他

    

选择"其他"时，学生必须填写具体运动名称，最多 32 个字符。学生开始运动后不能更换打卡类别；结束运动后可在提交前核对运动项目。



### 5\.4 健康安全提醒



每个账号首次进入打卡页面时展示健康安全提醒：



> 请根据自身身体状况适量运动。如感不适应立即停止，必要时及时就医。
> 
> 



学生点击"我知道了"后可继续操作，同一账号后续不重复弹出。



### 5\.5 开始运动



学生点击"开始运动"后：



1. 记录开始时间。

2. 尝试获取当前位置。

3. 页面进入"运动进行中"状态并持续显示已运动时长。

4. 学生离开页面或应用进入后台时，计时仍按开始时间持续计算。

5. 学生再次进入时恢复显示同一条进行中的运动。

    

位置信息用于辅助判断运动真实性，不是打卡成立的必要条件。学生拒绝定位、设备无法定位或室内信号不足时，记录标记为"未获取位置"，仍可继续运动和提交。



**运动过程中拍照**：



- 运动进行中，学生可随时使用相机拍摄照片，作为运动过程的记录。

- 运动过程中拍摄的照片**仅保存为本地草稿**，不即时上传至服务器。

- 这些草稿照片在结束打卡时可供学生选择，作为打卡凭证。

- 运动过程中不允许从相册选择已有照片——所有凭证必须通过相机实时拍摄。

    

### 5\.6 结束运动



学生点击"结束运动"后，根据实际时长处理：



- **不足 1 小时**：提示"运动时长不足 1 小时，本次不计入打卡"。本次计时结束，不占用当日成功打卡次数；学生在当日开放时段内可重新开始。

- **满 1 小时但不足 2 小时**：显示实际时长，并按 1 小时进入提交环节。

- **满 2 小时**：自动结束并按 2 小时进入提交环节。

    

学生主动放弃进行中的运动时，本次计时不形成成功记录，也不占用当日次数。



**结束后拍照**：



- 结束运动后，学生同样可使用相机拍摄照片，补充运动凭证。

- 结束后的拍照同样遵循"仅相机拍摄、不从相册选择"的规则。

- 运动过程中已拍摄的草稿照片与结束后拍摄的照片一并进入凭证选择环节。

    

### 5\.7 补充运动信息



计时结束并达到 1 小时后，学生补充以下内容：



- 核对打卡类别和运动项目。

- 其他运动可填写不超过 200 字的运动说明。

- 选择至少 1 个能够证明本次运动的图片或视频凭证：

    - 可从**运动过程中拍摄的本地草稿照片**中选择。

    - 也可在此环节**使用相机新拍摄**照片。

    - **不允许从相册中选择已有照片**——所有图片凭证必须通过相机实时拍摄。

- 查看本次计入学时、打卡日期和课程归属。

    

### 5\.8 提交确认



学生提交前看到本次记录摘要，并确认以下内容：



- 打卡类别正确。

- 运动项目正确。

- 开始和结束时间正确。

- 计入学时正确。

- 凭证数量符合要求。

- 提交内容真实有效。

    

提交成功后：



1. 显示"打卡已记录"。

2. 清除本次草稿和进行中状态。

3. 记录立即计入学生当前学期的对应学时类别。

4. 更新首页和打卡记录页的累计学时。

5. 当日不再允许提交第二条记录。

    

## 6\. 运动凭证规则



### 6\.1 数量与格式



- 每次打卡至少提交 1 个凭证。

- 最多提交 6 张图片。

- 最多提交 1 段视频。

- 图片支持 JPG、PNG、WebP、HEIC 和 HEIF。

- 视频支持 MP4 和 MOV。

- 每张图片不超过 8MB。

- 单个视频不超过 100MB。

    

图片和视频可同时提交，但总数量仍须遵守以上限制。



### 6\.2 凭证内容



- 凭证应能反映本次运动项目、运动场景或运动过程。

- 凭证不得包含与本次运动无关、侵犯他人隐私或违反校纪校规的内容。

- 教师可在期末审查时查看图片和播放视频。

- 视频在记录中显示缩略图、播放标识和时长，图片可放大查看。

    

### 6\.3 提交失败



- 凭证尚未全部完成提交时，打卡记录不视为成功。

- 网络中断或凭证提交失败时，已填写内容保留为本机草稿，学生恢复网络后重新提交。

- 重复点击提交不会产生多条成功记录。

- 提交失败不占用当日成功打卡次数。

    

### 6\.4 拍照来源规则



> **核心原则**：所有打卡凭证照片必须通过相机实时拍摄，从源头保证运动凭证的真实性。
> 
> 



|规则项|说明|
|---|---|
|拍照时机|运动过程中和运动结束后**均可**使用相机拍摄照片|
|相册选择|**不允许**从相册（系统图库）选择已有照片作为打卡凭证|
|过程中照片存储|运动过程中拍摄的照片**仅保存为本地草稿**，不即时上传至服务器|
|结束打卡时选择凭证|结束打卡时，可从以下来源选择照片作为打卡凭证：① 运动过程中拍摄的草稿照片；② 当前环节使用相机新拍摄的照片|
|草稿生命周期|草稿照片随本次打卡计时生命周期——如放弃本次运动或计时不足 1 小时，相关草稿照片自动清除|
|提交后清理|打卡成功提交后，本次关联的本地草稿照片自动清除|



> **设计意图**：禁止相册选择是为了防止学生使用旧照片、网络图片或他人拍摄的照片冒充本次运动凭证。所有照片必须通过相机在当前运动上下文中实时拍摄，确保凭证与运动过程的时间关联性。
> 
> 



## 7\. 草稿规则



- 计时结束后尚未成功提交的内容可保存为草稿。

- Android 和 iOS 在学生离开页面前自动保存草稿。

- Web 端由学生点击"保存草稿"，存在未保存内容时在离开页面前提醒。

- 草稿只保存在当前设备，不计入学时，也不占用每日次数。

- 学生再次进入时可继续完善并提交。

- 草稿提交时仍按开始运动日期检查每日次数和打卡日期范围。

- 学期打卡截止后，未提交草稿不能再转为正式记录。

    

**照片草稿**：



- 运动过程中拍摄的照片同样作为草稿的一部分，仅保存在当前设备本地。

- 草稿照片随本次打卡计时生命周期——如放弃运动或计时不足 1 小时，相关草稿照片自动清除。

- 结束打卡时，草稿照片与当前拍摄的照片一并供学生选择，共同作为打卡凭证。

- 打卡成功提交后，本次关联的所有本地草稿照片自动清除，释放设备存储空间。

- 草稿照片不在不同设备间同步。

    

## 8\. 打卡记录与学时展示



### 8\.1 学生记录页



学生可查看本人全部成功记录，每条记录展示：



- 运动项目。

- 打卡日期。

- 开始时间和结束时间。

- 实际运动时长和计入学时。

- 课程相关或其他运动标签。

- 关联体育课程。

- 运动说明。

- 凭证数量和可查看的凭证。

- 教师选择向学生公开的反馈。

    

### 8\.2 学时统计



学生首页分别展示：



- 课程相关运动已完成学时和剩余学时。

- 其他运动已完成学时、校队或社团已确认抵扣学时和剩余学时。

- 本学期总完成学时和总剩余学时。

    

学生成功提交后记录默认有效并立即计入统计，不显示"待审查""已通过"或"补充材料"等过程状态。



### 8\.3 教师处理后的展示



- 记录被标记为无效后，从有效学时中扣除，并在学生记录页显示"不计入学时"。

- 记录恢复有效后，学时重新计入。

- 记录学时被修正后，首页和记录页同步显示修正结果。

- 教师选择公开反馈时，学生可在对应记录中查看。

- 教师仅添加内部备注时，学生端不显示该内容。

    

## 9\. 教师审查流程



### 9\.1 审查时间



- **学期中抽查纠偏**：教师在学期中可随时查看学生记录，对明显异常（如虚假凭证、代打卡等）进行抽查标记。学生提交后记录默认有效并计入学时，教师抽查不影响正常打卡流程。

- **截止后集中复核**：学期打卡截止后，任课教师必须完成对本人课程学生全部记录的最终集中复核。

    

### 9\.2 查看学生概览



教师先查看班级学生概览，包括：



- 学号和姓名。

- 总打卡时长和打卡次数。

- 课程相关运动时长和其他运动时长。

- 当前有效记录数量。

- 已标记无效的记录数量。

- 最近一次打卡日期。

    

### 9\.3 查看学生详情



教师进入单个学生详情后，可按日期查看全部图片、视频和对应打卡记录，也可按时间倒序查看记录列表。每条记录提供运动项目、日期、学时、说明、位置状态和凭证。



### 9\.4 标记无效



教师发现虚假、作弊、内容不符或凭证无法证明本次运动的记录时：



1. 选择一条或多条记录。

2. 标记为无效。

3. 填写处理原因或内部备注。

4. 记录不再计入学生有效学时。

5. 必须向学生发送站内通知（不可静默），提供异议入口。

    

### 9\.5 恢复有效



教师复核后确认记录有效时，可将记录恢复为有效。恢复后记录重新计入学生学时，并同步更新学生首页和记录页。



### 9\.6 修正学时



教师可依据记录的实际开始时间、结束时间和凭证修正计入学时。修正结果须遵守以下规则：



- 不足 1 小时不计入学时。

- 满 1 小时但不足 2 小时计 1 小时。

- 满 2 小时计 2 小时。

- 单日最终计入学时不超过 2 小时。

    

### 9\.7 备注与学生反馈



- **内部备注**仅供负责该课程的教师查看，用于记录审查依据和后续观察事项。

- **学生反馈**向学生本人展示，用于说明记录处理结果和后续注意事项。

- 教师添加内容时明确选择受众，默认使用内部备注。

    

### 9\.8 大量异常记录



同一学生出现大量虚假或异常记录时，教师可批量标记无效并联系体育部管理员。管理员核实后可限制该学生继续打卡，处理结果由任课教师同步给学生。



## 10\. 异常与边界场景



### 10\.1 当日已有成功记录



学生再次开始或提交时，显示"今日已打卡"，并引导查看当日记录。即使当日已提交记录仅计 1 小时，也不能再次提交。



### 10\.2 跨日运动



23:30 开始、次日 00:30 结束的运动按开始日期记录，并计入开始日期的每日次数。结束日期仍保留自己的打卡次数。



### 10\.3 定位不可用



定位失败不阻止计时和提交。记录保留"未获取位置"的事实，教师结合其他内容进行期末审查。



### 10\.4 应用退出或设备重启



进行中的运动以已记录的开始时间持续计算。学生重新进入后恢复进行中状态；达到 2 小时时按 2 小时上限处理。



### 10\.5 更换设备



学生在另一设备进入时可看到已有进行中的运动，但不能重复开始。学生应回到原设备结束和提交；原设备不可用时，可放弃本次计时后重新开始。



### 10\.6 无当前体育课程



学生没有当前学期在读体育课程时，不能发起课程相关运动，也不能提交由任课教师负责的其他运动记录。页面引导学生先完成选课或联系体育部处理课程归属。



### 10\.7 凭证不合规



凭证数量、格式或大小不符合规则时，学生不能提交。页面明确指出具体文件和原因，其他已选择文件保持不变。



### 10\.8 教师期末处理导致学时不足



教师将记录标记无效或修正学时后，学生最终有效学时可能低于学期要求。学生记录页展示最新有效学时和公开处理说明，教师按课程成绩管理规则处理未达标结果。



## 11\. 隐私与安全



- 首次登录时，学生同意隐私政策后方可使用打卡功能。

- 位置信息只用于辅助判断运动真实性，不用于持续跟踪学生。

- 位置信息保留 90 天，超过期限后教师端显示"位置信息已过期"。

- 学生只能查看自己的记录和凭证。

- 任课教师只能查看其负责课程学生的记录和凭证。

- 通知内容不直接展示姓名、成绩或运动凭证等个人信息，学生登录后查看详情。

- 学生提交的运动内容和凭证必须遵守学校隐私保护和行为规范。

    

## 12\. 完整业务闭环



```Plain Text
学生进入打卡页
  → 检查账号、时间范围、当日次数、进行中记录和课程归属
  → 选择课程相关运动或其他运动
  → 选择运动项目
  → 开始运动并记录开始时间
  → 运动进行中（可随时拍照，照片仅存本地草稿）
  → 学生结束或满 2 小时自动结束
  → 根据实际时长换算 0、1 或 2 小时
  → 达到 1 小时后补充说明和凭证（从草稿照片中选择 + 可新拍摄，禁止从相册选择）
  → 学生确认并提交
  → 成功记录立即计入对应学时
  → 学生查看记录和累计进度
  → 学期打卡截止
  → 任课教师集中审查
  → 保持有效、标记无效、恢复有效或修正学时
  → 学生查看最终有效学时和公开反馈
  → 结果进入课程成绩处理流程
```

## 7\. 课程管理



### 7\.1 业务概述



系统不设课程任务模块。学生一学期只能选择一门体育课。自主其他运动由该门体育课的授课老师统一负责。



### 7\.2 学生选课与退课（Add/Drop）



一个学生、一个学期，最多只能有一门在读状态的 PE 课。这是整个选课设计的基础规则，通过系统校验保障。



#### 场景一：学生正常退课（Drop）



退课操作由教师在 Web 端执行，或通过 CSV 导入标记退课。



退课发生时，系统做以下处理：



- 该课程的学生选课状态从"在读"变为"已退课"，记录退课时间

- 学生的打卡记录全部保留（退课前的运动记录是真实发生的）

- 学生的学时进度数据保留

- 教师的审查记录保留

    

退课后，学生端的效果：



- 该课程从「当前课程」移到「历史课程」

- 状态显示"已退课"

- 不能再对该课程提交打卡

- 历史打卡记录可查看（只读）

    

退课后的数据归属规则：



- 校队/社团抵扣：跟着课程走，不退不转。抵扣是授课老师为当前课程确认的，换课后新老师需要重新确认

- 已打卡时长：保留在历史记录中，不转移到新课

- 成绩数据：保留在旧课程中，状态标记为"已退课"，不计入新课成绩

    

退课后的成绩显示：



- 当前课程（如有新课程）：成绩正常显示

- 历史课程：旧课程显示"已退课"，成绩数据保留但标注"\(退课\)"

- 成绩单：仅显示最终完成的课程成绩，退课课程不出现

    

#### 场景二：学生退课后加入另一门课



学生一学期内可以退课换课。举例：学生先选了 GEPE101，退课后又选了 PEB203。



初始状态：学生选了 GEPE101（Section 1004），已打卡 8h。



退课操作：教师对 GEPE101 执行 Drop。该课程状态变为"已退课"，打卡记录保留（8h），成绩数据保留。退课时间记录为 3 月 15 日。



加入新课：教师对 PEB203 执行 Add。系统校验该生当前学期无其他在读课程（GEPE101 已退课，校验通过）。PEB203 状态变为"在读"，选课时间记录为 3 月 16 日，打卡时长从 0 开始。



最终状态：GEPE101（已退课，3 月 15 日退课）打卡 8h，只读历史；PEB203（在读，3 月 16 日加入）打卡 0h，当前课程，可正常打卡。



关键规则：两门课完全独立。GEPE101 的 8h 打卡不转移到 PEB203，换课后从零开始。学生端课程列表中 PEB203 在「当前学期」，GEPE101 在「历史课程」。



#### 选课状态流转



教师添加学生后，该生在该课程的状态为"在读"。同一学生、同一学期，最多只有一条"在读"记录。



教师退课后，状态变为"已退课"。学生可再选新课。



学期结束归档后，状态变为"已完成"。数据只读。



### 7\.3 课程生命周期



管理员创建课程 → 分配授课老师 → 设定学期 → 教师导入学生名单（自动建号并发送邮件）→ 学生进行运动打卡（课程运动 \+ 自主其他运动）→ 教师录入考试成绩、体测成绩、平时分 → 计算总成绩 → 课程归档 → 学期归档。



### 7\.4 学生端课程查看



学生可查看当前学期的体育课和历史已修的体育课。



**当前学期课程**（学生一学期仅有一门）：



- 课程信息：课程代码、Section、名称、教师

- 课程进度：课程运动打卡时长完成情况

- 课程记录：该课程下的打卡记录

    

**历史课程**（按学期倒序展示过往所有已修体育课）：



- 历史课程列表

- 课程信息：课程代码、Section、名称、教师、所属学期

- 课程成绩：该历史课程的最终成绩（如已归档）

- 课程记录：该历史课程下的打卡记录（只读回溯）

    

历史课程数据来源于学生真实的选课记录。历史课程仅供查看，不可进行打卡操作。当前学期课程与历史课程分区展示，可折叠。



### 7\.5 教师端课程管理：名单导入与账户体系



管理员不参与账号创建。教师导入课程名单时，系统自动完成所有账号创建工作。



#### 整体流程



教师从学校选课系统导出 CSV → 导入系统（列映射自动匹配，记住偏好）。



对于 CSV 中的每个学生：



- 已有账号的学生（老生）：更新个人信息（姓名、学院、班级等），直接关联课程，不重置密码，不发送邮件

- 首次出现的学生（新生）：自动创建账号，发送密码设置邮件到学校邮箱。邮箱地址统一为学号加学校邮箱域名

    

#### 密码设置流程



不设默认密码，每个学生通过学校邮箱独立设置密码。



新生收到密码设置邮件（7 天有效），点击邮件中的链接进入密码设置页面，设置密码后即可登录。



老生忘记密码时，在登录页点击"忘记密码"，输入学号后系统发送重置邮件（30 分钟有效）。



#### 邮件通知



新生密码设置邮件：



- 发件人：`noreply@bnbu.edu.cn`

- 收件人：学生学校邮箱

- 主题：BNBU 体育打卡系统 \- 请设置你的账号密码

- 正文：包含学生姓名、课程信息、授课老师、密码设置链接（7 天有效）、学号

    

忘记密码邮件（老生）：



- 主题：BNBU 体育打卡系统 \- 密码重置

- 正文：包含密码重置链接（30 分钟有效）

    

#### 密码设置页面（Web / App 通用）



学生点击邮件链接后进入密码设置页面，页面显示学号，提供新密码和确认密码输入框。密码要求至少 8 位，含字母和数字。点击确认后，系统验证 Token，写入密码，标记 Token 已使用，跳转登录页。



#### 列映射——导入体验



不同教师从学校选课系统导出的 CSV 列名不同。系统提供两项能力：自动匹配 \+ 记住偏好。



导入时，系统展示列映射界面，左侧为系统字段（学号、姓名、学院、班级、课程代码、Section、选课状态），右侧为 CSV 列头。系统自动识别并匹配列名，匹配成功的显示确认标记，未识别的列教师可手动从下拉框选择。



匹配规则：系统根据关键词命中程度进行自动匹配。精确匹配或关键词命中的列自动填入，未识别的列教师手动选择。



偏好记忆：每位教师的列映射偏好被系统记录。首次导入时依靠自动匹配加手动调整；再次导入时优先匹配该教师上次保存的 CSV 列头名称。CSV 格式变化时教师可手动调整，系统会更新偏好。



#### 预览导入与批量确认



点击「预览导入」后展示变更摘要：



- 新增：新生，将创建账号并发送邮件

- 已有：老生，直接关联课程

- 退课：CSV 中选课状态标记为"退课"的学生

- 冲突：该生已在同学期另一门 PE 课中在读，拒绝导入并提示"已选其他 PE 课"

    

教师点击「确认导入」后系统执行：新生创建账号；老生关联课程；退课学生标记退课；新生邮件加入发送队列。



#### 边界情况处理



教师上传了错误 CSV 的处理：预览页点取消，不执行任何操作。已确认的导入可通过退课逐个撤销。



新生 7 天未点击邮件链接的处理：Token 过期后，学生在登录页使用「忘记密码」功能重新获取邮件。



CSV 中缺少邮箱列的处理：不依赖 CSV 中的邮箱，统一使用学号加学校域名生成邮箱地址。



邮件发送失败的处理：不阻塞导入流程，后台重试 3 次。教师端可查看邮件未送达的学生列表。



同一学生出现在两个教师 CSV 的处理：第一个导入成功，状态为在读；第二个导入检测到冲突，拒绝导入并提示"已选其他 PE 课"。



### 7\.6 管理员端课程管理



管理员可创建、编辑、删除课程。



课程创建必填字段：课程代码（学校规定的课程编号）、Section（分班编号）、课程名称（中文全称）、授课老师（下拉选择）、所属学期（下拉选择）、课程容量（最大学生数）。



课程编码规则：课程代码加 Section 组成唯一标识（如 GEPE101\-1004），由管理员创建时手动填写，系统不自动生成。



课程状态流转：草稿 → 已发布 → 进行中 → 已结课 → 已归档。草稿状态可删除；进行中状态不可删除（已有学生数据）。



课程复制：管理员可将上学期课程复制到新学期，保留课程代码、名称、授课老师。Section 和学期需重新指定，学生名单从零开始。

## 12\. 学期管理



### 12\.1 业务概述



管理员管理学期。学期锁定后阻止打卡和成绩修改。



### 12\.2 学期属性



- 名称（如"2026 Spring"）

- 日期范围（如"2026\.02\.24 \- 2026\.06\.28"）

- 状态：进行中、草稿、已归档

- 锁定：是或否。锁定后阻止打卡和成绩修改，除非解锁

    

### 12\.3 学期切换流程



管理员创建新学期：设置学期名称和日期范围；设置学时规则（默认继承上学期配置，可修改）；设置耐力跑换算表（默认沿用最新版本）。



管理员创建新学期课程：可手动创建，也可从上学期复制课程结构。



旧学期到期后管理员锁定：锁定后禁止打卡；锁定后禁止修改成绩（除非解锁）；教师可继续提交归档。



归档完成后状态变为"已归档"：学生端当前学期无课程时显示历史课程；数据只读。



关键规则：



- 同一时间最多只有一个学期处于"进行中"状态

- 学期的"锁定"和"归档"是两个独立操作：锁定阻止打卡和成绩修改，归档是最终确认

- 历史学期数据永久保留，学生可随时查看

- 学期归档后，该学期所有数据变为只读

    

学生端学期切换逻辑：学生登录后查询当前学期是否有在读课程。有则首页展示当前学期课程和打卡入口；无则首页展示"暂无当前学期课程"，可下拉查看历史课程。

## 1\. 模块总览



### 1\.1 本模块做什么



本模块完成体育课从"成绩规则配置"到"成绩归档"的闭环：



1. 教师配置并**发布成绩规则**（学生据此知悉本课成绩构成与打分方式）

2. 学生进行运动打卡（打卡与成绩规则完全独立，规则未发布时也可正常打卡）；教师审查打卡（标无效、调时长、补录学时、写意见）

3. 系统按有效学时生成体育打卡分；教师录入专项考试、平时、体测等

4. 教师发布成绩 → 公示与申诉 → 管理员发起并完成课程归档

5. 全部课程归档完成后，管理员执行学期归档

    

**关键规则**：打卡与成绩规则完全独立。学生打卡不受成绩规则发布状态影响——成绩规则未发布时学生仍可正常提交打卡。成绩规则仅影响学生能否查看本课成绩构成与打分方式。



### 1\.2 端到端主流程



```Plain Text
教师配置成绩规则草稿（项目、权重）
        │
        ▼
教师"发布成绩规则"（结构与权重冻结）
  → 学生收到通知，可查看本课成绩构成
        │
        ▼
学生持续打卡（时长即时计入，与成绩规则发布状态无关）
教师随时审查打卡（标无效 / 调时长 / 补录 / 意见）
  → 学生有效学时与达标进度实时更新
        │
        ▼
系统按有效学时自动生成"体育打卡"分
教师录入专项考试、平时、体测等
        │
        ▼
教师"发布成绩" → 进入公示期
  → 学生查看明细、提交申诉
  → 教师复核（采纳改分 / 驳回）
  → 教师驳回即为最终结果
        │
        ▼
公示期满且无未处理申诉
        │
        ▼
管理员发起课程归档（系统预检）
  → 管理员批准 / 退回
        │
        ▼
本学期全部课程均已归档
        │
        ▼
管理员执行"学期归档"（另一名管理员确认）
```



### 1\.3 本模块术语



|术语|含义|
|---|---|
|有效学时|计入达标统计的学时：有效打卡记录的调整后时长 \+ 教师补录学时；不含已标无效的记录|
|目标学时|学生本学期需完成的学时要求；已确认校队/社团抵扣后，按抵扣后的剩余目标计算|
|体育打卡分|由有效学时按公式自动折算得到的百分制分数，作为成绩项目之一|
|发布成绩规则|教师将本课成绩项目与权重定版；定版后不可再改项目结构与权重|
|发布成绩|教师将成绩对学生可见并进入公示期|
|课程归档|单门课成绩提交体育部审核并获批准后的最终确认状态|
|学期归档|本学期所有课程均已课程归档后，管理员对整个学期的最终确认|
|未录入|教师尚未对该成绩项给出任何结论（无分数、未标缺考）|
|缺考|教师明确标记学生未参加该考试项；计 0 分参与加权|

### 1\.4 课程成绩状态



一门课在本模块内按以下状态推进（不可跳步）：



|状态|含义|教师可做|学生侧|
|---|---|---|---|
|规则未发布|尚未定版成绩项目与权重|编辑规则草稿|可正常打卡；不可见分项成绩；可见"请等待教师发布成绩规则"|
|打卡与录入中|规则已发布；学生打卡与教师录分并行|审查打卡；录入/修改分数、标记缺考|可正常打卡；可查看成绩构成、有效学时进度；不可见正式总分明细|
|公示中|已发布成绩，公示期内|处理申诉、按申诉结果改分|可见分项与总分，可申诉；打卡是否仍开放以学期打卡时间窗为准|
|待提交归档|公示期满且无未处理申诉|等待管理员归档|成绩只读|
|归档审核中|管理员发起归档审核|等待；被退回后回到"打卡与录入中"或"待提交归档"（视是否需改分）|成绩只读；打卡是否开放以学期打卡时间窗为准|
|已退回|管理员退回|修正成绩或材料后等待管理员再次归档|成绩只读|
|已归档|管理员批准|不可直接改分；仅能走成绩修正申请|成绩只读|
|修正审批中|归档后申请改分|等待管理员审批|仍显示归档成绩，批准后更新|



学期锁定后：禁止改分；已处于"待提交归档"或之后的流程可继续完成归档。

## 2\. 教师打卡审查



### 2\.1 业务定位



- 学生提交打卡后，时长立即计入统计，**不经过**教师逐条"通过/驳回"。打卡与成绩规则完全独立，成绩规则未发布时学生仍可正常打卡。

- 教师对打卡做**抽查式审查**：标记无效、调整时长、补录学时、填写意见。

- 审查从规则发布、学生开始打卡起即可进行，不限于学期末；操作后学生有效学时与达标进度**立即更新**。

- 不提供：逐条通过/驳回/补材料、AI 查重、审核状态标签（待审核/已通过等）、热力图、待审核数 KPI。

    

### 2\.2 班级打卡总览（两层）



#### 第一层：学生打卡概览表



|列|说明|
|---|---|
|学号|学生学号|
|姓名|学生姓名|
|申报总时长|学生提交记录的申报时长合计（含随后被标无效的）|
|有效总时长|当前计入达标的有效学时（含补录）|
|打卡次数|本学期提交条数|
|有效打卡次数|打卡次数 \- 已标无效条数|
|课程运动有效时长 / 自主运动有效时长|两个学时桶分别统计|
|最近打卡日期|最近一次提交日期|
|无效条数|已标无效的记录数|
|重点关注|是否命中重点关注条件（见 2\.6）|



名单范围：本课当前 enrolled 学生。已退课学生不在本表主列表，另有只读历史区，不进入归档名册。



#### 第二层：单个学生详情（双栏）



**左侧**：姓名、学号、学院、年级；有效总时长、打卡次数、无效条数；课程运动 / 自主运动有效时长与目标学时（抵扣后）；教师对该生的课程级备注



**右侧 — 模式一：凭证相册（只读预览）**：凭证按 年→月→日 分组堆叠；视频显示缩略图、播放按钮、时长；点击凭证进入该次打卡详情；本模式**不能**批量勾选标无效



**右侧 — 模式二：打卡记录列表（可操作）**：按时间倒序；每条显示运动类型、开始日期、申报时长、调整后时长、有效/无效状态、凭证缩略图、描述；支持多选，执行批量标无效/恢复有效



### 2\.3 标记无效 / 恢复有效



**操作**：在列表模式勾选记录 → 选择"标记为无效"，必选原因：虚假凭证/代打卡/时长不符/其他（选其他须填说明）→ 必须向学生发送 App 内通知（不可静默），提供异议入口 → "恢复为有效"可撤销无效标记



**效果**：无效记录不计入有效学时、不计入有效打卡次数；学生对无效记录可见，状态显示为"无效"及原因；有效学时变化后达标进度立即更新



### 2\.4 调整单条打卡时长



- 教师调整后的有效记录时长**只能是 1 小时或 2 小时**（与打卡步进规则一致）

- 若需将某条记录学时变为 0：使用\*\*「标记无效」\*\*操作，不可通过"调整为 0"实现

- 调整后以新时长计入有效学时（若该条为无效，则仍不计入，直至恢复有效）

- 每次调整记录操作人、时间、旧值、新值；必须通知学生（不可静默），提供异议入口

- 调整时长**不受**"每日最多申报 2h"限制约束（审查纠偏优先于日常额度）

    

### 2\.5 学时补足（无效或不足之后）



两条通道并行，均计入有效学时：



|通道|谁操作|规则|
|---|---|---|
|自行补打|学生|打卡时间窗开放、且未触发账号限制时，按每日 1 次、每日最多 2h、每次 1h 或 2h 的日常规则继续提交|
|教师补录|授课老师|直接为学生增加课程运动或自主运动学时；必填原因；可附证明材料；**不占用**学生"每日 1 次 / 每日 2h"额度|



### 2\.6 重点关注名单



系统按规则自动标记，供教师优先审查（不替代人工结论）：



- 有效学时未达目标学时的 50%（学期过半后）

- 打卡截止前 7 天内集中提交超过本学期有效学时一半

- 全部有效记录均无定位信息

- 无效条数 ≥ 3

    

### 2\.7 意见（两类）



|类型|可见范围|用途|
|---|---|---|
|仅内部备注|本课授课老师|审查备忘，如"疑似代跑"|
|反馈给学生|该学生|告知问题，如"缺少场地照片"|



添加时必选类型，默认"仅内部备注"。反馈给学生的意见在学生端打卡记录中展示。



### 2\.8 作弊与限制打卡



```Plain Text
发现疑似作弊
  → 标记相关记录无效（选原因）
  → 添加内部备注
  → 通知学生（不可静默）
  → 若需限制该生继续打卡：向管理员发起"限制打卡"申请
       → 管理员在用户管理中执行限制 / 解除
```

## 3\. 成绩规则



### 3\.1 默认模板与自定义



每门课开课成绩规则时，系统提供**学校默认四块模板**：



|项目类型|默认名称|说明|
|---|---|---|
|体育打卡|体育打卡|由有效学时自动折算，教师可手工覆盖单人分数|
|专项考试|专项考试|下挂若干考试子项|
|平时表现|平时表现|按课次出勤 \+ 扣分规则 / 手工分|
|体测|体测|含耐力跑自动换算及其他体测分|



规则：上述四种**类型不可删除**；名称可改；权重可改；教师可**新增**自定义成绩项（类型为"自定义"），仅支持手工百分制录入；所有成绩项权重之和必须等于 100%；耐力跑只归属"体测"，不作为专项考试子项。



### 3\.2 专项考试子项



- 教师在"专项考试"下增删改子项（如投篮、绕杆、立定跳远技能考等）

- 每个子项录入**百分制分数**（0\-100）

- 专项考试项得分 = 各子项算术平均（教师也可指定子项权重，子项权重和为 100%）

    

### 3\.3 规则版本与发布



- **发布前**：教师只编辑规则草稿；学生可正常打卡（打卡与成绩规则独立）

- **发布后**：成绩项目结构与权重冻结（不可增删项目、不可改权重、不可改项目类型）；学生可查看本课成绩构成；教师可审查打卡并录入修改分数直到"发布成绩"

- 每次发布产生版本号；历史版本可查看

- 若必须改结构：教师向管理员申请废止当前规则并发布新版本；管理员批准后已录入分数按项目名称自动迁移

    

### 3\.4 体育打卡分折算



```Plain Text
体育打卡分 = min(100, 有效学时 / 目标学时 * 100)
```



- 目标学时 = 学时规则要求 \- 本课已确认的校队/社团抵扣时长

- 结果保留一位小数，参与总分前再按总分规则取整

- 教师可对单个学生**手工覆盖**体育打卡分；覆盖后该生不再随学时自动变，直至教师取消覆盖

## 4\. 成绩录入



### 4\.1 录入总则



- 仅本课授课老师可录入

- 支持逐人录入、保存草稿、按模板批量导入

- 批量导入：学号必须属于本课当前在册学生；分数 0\-100；导入前展示预览，确认后写入

- 退课学生不出现在录入主表，其历史分数只读保留，不进正式归档导出

    

### 4\.2 未录入 / 缺考 / 0 分



|状态|如何产生|展示|是否计入总分|能否发布成绩|
|---|---|---|---|---|
|未录入|默认；教师未操作|"未录入"|不计入（此时不可发布）|否（任一必填项未录入则禁止发布）|
|0 分|教师录入 0|"0"|按 0 加权|是|
|缺考|教师显式"标记缺考"|"缺考（计 0）"|按 0 加权|是|



### 4\.3 平时表现 / 课次出勤



- 教师设定本学期课次列表（课次序号 \+ 日期）；可批量生成

- 每个在册学生、每个课次，出勤状态为：出勤/迟到/请假/缺勤

- **本系统默认：未记状态在计算平时分时视为出勤**

- 扣分规则：设定基数分（默认 100）；设定迟到、请假、缺勤每次扣分值；设定平时分下限（默认 0）

- 自动平时分 = max\(下限, 基数 \- 各状态扣分总和\)

- 教师可手工覆盖最终平时分，取消覆盖后恢复自动计算

    

### 4\.4 体测录入



- 身高、体重 → 自动算 BMI（展示用）

- 耐力跑用时 → 按换算表换算为百分制

- 其他体测项目：教师直接录入**百分制分数**

- 体测项总分默认 = 耐力跑换算分与其他已录入体测子项的算术平均；教师可配置子项权重

- 耐力跑免测通过：该子项显示"免测"，**由教师为每名免测学生自定义分数**（不再固定为 100 分），教师端禁用该生耐力跑用时录入

    

### 4\.5 专项考试录入



- 按子项录入百分制；支持标记缺考

- 缺考计 0 分参与专项考试项加权计算

## 5\. 成绩公示与申诉



### 5\.1 发布成绩



前置条件：成绩规则已发布；所有在册学生、所有成绩项均非"未录入"；体育打卡分已生成或已手工覆盖。



操作：教师点击"发布成绩"→ 课程进入"公示中"→ 学生收到成绩发布通知（不可关闭该类通知）。



### 5\.2 公示期



- 默认 **5 个自然日**，自发布成功时刻起算

- 体育部管理员可按课程将公示天数配置为 **1\~7 个自然日**

- 公示期结束且无未处理申诉 → 进入"待提交归档"

    

### 5\.3 申诉



**学生**：可对任意已发布的成绩项目申诉；须选争议项目、填写理由（\>=10字）；同一项目存在未处理申诉时不可再提



**教师复核**：采纳则修改成绩并填写说明；驳回则填写驳回理由。**教师驳回即为最终结果，不可升级。**



**教师处理时限**：申诉提交后 48 小时内教师须处理；超时未处理系统自动提醒教师。



### 5\.4 公示期内改分与学时变化



- 因申诉采纳或公示期内学时变化导致体育打卡分重算：**不延长公示期**。改分后学生可即时查看更新后的分数

- 已进入"待提交归档"或之后：学时审查变更不再自动改分，须走归档后成绩修正

    

### 5\.5 归档后成绩修正



```Plain Text
教师提交修正申请（课程、学生、项目、旧值、新值、原因）
  → 管理员审批
  → 通过后写入新分，课程保持"已归档"
  → 记录审计（操作人、审批人、旧值→新值）
  → 通知该学生与任课教师
```

## 6\. 耐力跑成绩换算



### 6\.1 用途划分



|端|用途|
|---|---|
|学生端|**试算工具**：输入分钟\+秒，查看换算分与等级；**不写入**正式成绩|
|教师端|**正式录入**：录入用时后写入体测耐力跑子项分数；另提供与学生相同的试算工具|
|管理端|维护换算表版本并启用|



### 6\.2 换算逻辑（本校标准）



- 按**性别 \+ 年级组**查表，共 4 套：男\-大一大二、男\-大三大四、女\-大一大二、女\-大三大四

- 女生 800m，男生 1000m，由性别自动决定

- 用时落入表中某秒对应分数（0\-100），并映射等级：优秀/良好/及格/不及格

- 越界：用时短于表内最快档→记满分；用时长于表内最慢档→记 0 分

    

### 6\.3 换算表管理



- 系统初始化时载入标准换算表作为版本 1

- 管理员可编辑、批量更新、复制为新版本；同时只有一个"启用中"版本

- **新录入**一律用当前启用版本

- **未归档**课程：教师可对本课执行"按当前启用表重算"

- **已归档**课程：不因换算表变更而重算

## 8\. 成绩展示与导出



### 8\.1 学生端



- 仅查看，不可改分

- 总分公式：round\(项目得分 \* 权重%\) 之和，四舍五入取整

- 及格线：总分 \>= 60 显示"及格"，否则"不及格"；重修、清考不在本模块处理

    

### 8\.2 教师端



- 成绩总览：在册学生 \* 各成绩项 \* 总分 \* 及格标记 \* 缺考标记

- **自定义导出**：自选列，下载 Excel，**不**触发归档

- **提交课程归档**：生成正式成绩快照并进入管理员审核队列

    

### 8\.3 正式归档导出列



管理员维护"教务导出模板"（列顺序与列名）。课程归档批准后的正式文件按该模板输出，固定包含：学号、姓名、课程代码、Section、学期；各成绩项得分或状态；总分、及格标记。

## 9\. 课程归档与学期归档



### 9\.1 三层关系



学期归档（管理员，全学期最终确认）下辖课程 A/B/\.\.\. 归档（**管理员独立发起并完成**，不再经由教师提交）



### 9\.2 课程归档



**发起者**：管理员（体育部管理员或超级管理员）独立发起课程归档，不依赖教师提交。



**前置条件**：课程状态为"待提交归档"；公示已结束；无未处理申诉



**预检项目**：有效学时未达目标学时（阻断，除非强制提交）；体测必填子项缺失（阻断，除非强制提交）；存在未录入项（阻断，不可强制）；存在未处理申诉（阻断，不可强制）；存在缺考（警告，不阻断）；成绩规则未发布（警告，不阻断）



**归档操作**：管理员在课程列表中选择待归档课程 → 系统执行预检 → 通过后确认归档 → 成绩锁定，生成正式导出快照



### 9\.3 管理员对课程归档的处理



|操作|结果状态|说明|
|---|---|---|
|批准|已归档|成绩锁定；生成正式导出快照|
|退回|已退回|教师修正后管理员重新发起归档；通知教师|
|提醒教师|保持待提交归档|仅发催办通知，不改状态|



已归档课程**不能**再"退回"；纠错一律走成绩修正申请。



### 9\.4 学期归档



**前置条件**：本学期所有课程均为"已归档"；无进行中的成绩修正申请



**操作**：管理员 A 发起学期归档 → 另一名管理员 B 确认（A、B 不可为同一人）→ 学期归档完成，本学期成绩全面只读



若活跃管理员不足两人，系统拒绝发起并提示需至少两名管理员。

|对方业务|本模块如何衔接|

## 9\. 免测申请与审核



### 9\.1 业务概述



学生可申请两类免除：



1. **体测免测**: 800m / 1000m 耐力跑免测

2. **校队/社团免打卡**: 校队或社团成员申请免打卡

    

### 9\.2 学生端免测流程



#### 体测免测（800m / 1000m）



```Plain Text
学生提交申请 → 选择免测类型(800m/1000m，根据性别自动默认) →
  填写理由 → 上传证明（至少1张图片） → 提交 →
    教师审核 → 通过/驳回
    驳回后 → 补充材料重新提交
```



**三端统一**:



- 免测类型: 800m（女生）/ 1000m（男生）

- 理由: 必填

- 证明: 至少上传 1 张图片（如医院证明）

- 重复检查: 同类型不能有重复待审核申请

- 驳回后可补充材料重新提交

    

#### 校队/社团免打卡



```Plain Text
学生提交申请 → 选择类型(校队/社团) → 填写组织名称 →
  填写理由 → 上传证明材料 → 提交 →
    授课老师审核 → 通过/驳回
    通过后 → 教师自定义抵扣时长 + 选择抵扣类型
```



**三端统一**（新增 Web 端和 iOS 端支持）:



- 类型: 校队免打卡 / 社团免打卡

- 组织名称: 必填（\<=128 字）

- 理由: 必填

- 证明: 上传身份证明材料

- **Web 端新增**: 校队/社团免打卡申请入口

    

### 9\.3 免测/抵扣的时效与撤销



**体测免测**:



- 一旦批准，**本学期内不可撤销**。体测是学期中一次性完成的，批准后学生已不需要参加体测

- 如果学生康复后主动要求恢复体测 → 需向教师提交申请，教师手动将免测状态改回"未申请"

- **免测学生分数由教师自定义**：教师为每名免测学生单独设定体测耐力跑子项分数（不再固定为 100 分），最大化兼容各种实际场景

    

**校队/社团抵扣**:



- 抵扣确认后，如果学生中途退出校队，**不自动撤销**（学生退出前确实参加了训练）

- 如果学生主动告知教师"该生已退出" → 教师可手动撤回抵扣

- 撤回抵扣后，student\_progress 中的抵扣值清零，学生需重新打卡补足差额

    

**免测申请提交次数**: 最多 **2 次**（首次 \+ 1 次补充材料）。第 2 次被驳回后不可再次提交，需联系教师线下处理。



### 9\.4 教师端免测审核



教师有两个审核入口:



|审核类型|决策|说明|
|---|---|---|
|体测免测审核|通过 / 驳回|审核学生提交的 800m/1000m 免测申请|
|校队/社团免打卡审核|通过 / 驳回|审核通过后，教师可自定义抵扣时长|



> **修复**: 教师使用体测免测审核和耐力跑换算时，后端需支持教师 Token 调用（先前版本教师 Token 被 requireRole\("student"\) 拒绝）。
> 
> 



---



## 10\. 校队/社团认证与学时抵扣



### 10\.1 业务概述



学生参加校队或社团可获得打卡时长抵扣。授课老师在确认抵扣时，可**自行设置抵扣数值**，并可**自行选择抵扣哪一类时长**（课程运动 或 自主其他运动，或两者各抵扣一部分）。学生端同步展示**扣除抵扣后需要完成的实际打卡时长**。



> **核心变更**:
> 
> 
> 
> - 抵扣时长数值由授课老师**自定义设置**，无固定数字
> 
> - 抵扣目标由教师**自行选择**：可抵扣课程运动、自主其他运动，或两者各分配一部分
> 
> - 学生端根据抵扣结果，动态展示**还需完成的总打卡时长**
> 
> 



### 10\.2 认证生命周期



```Plain Text
学生提交校队/社团免打卡申请 →
  授课老师审核 + 自定义抵扣时长 + 选择抵扣类型 →
    学时记入 student_progress →
      学生端同步更新需完成的剩余打卡时长
```



### 10\.3 各角色操作



#### 学生



- 查看自己的组织认证状态和抵扣详情

- 查看**扣除抵扣后还需完成的实际打卡时长**（课程运动还需 Xh，自主其他运动还需 Yh）

- 首页和学习进度中展示的是抵扣后的目标值

- 状态: 待确认 / 认证有效

    

#### 授课老师



- 查看待确认的组织抵扣

- **自定义抵扣时长**: 教师手动输入抵扣的时长数（无固定数字，上限 20 小时）

- **自定义抵扣类型**: 教师可选择将抵扣时长分配到课程运动、自主其他运动，或两者各分配一部分

- 确认抵扣 / 拒绝抵扣

- 手动撤回抵扣（学生退出校队/社团时）

    

#### 管理员



- 全部组织认证记录管理

- 筛选: 全部 / 待确认 / 已生效 / 已阻止

- 操作: 确认生效、强制抵扣、撤销抵扣

    

### 10\.4 抵扣规则



- 抵扣时长数值由教师**自定义设置**（上限 20 小时，含课程运动与自主其他运动合计）

- 抵扣类型由教师**自行选择**：可抵扣课程运动、自主其他运动，或两者各分配一部分

- **课程运动与自主其他运动均可被校队/社团认证抵扣**，合计最多抵扣 20 小时

- 抵扣后学生端**同步展示**需完成的总打卡时长（课程运动还需 Xh，自主其他运动还需 Yh）

- 抵扣确认后中途退出校队：**不自动撤销**，需教师手动操作

- **社团负责人不参与其中**，只需要老师审核入社/入校队文件即可（已删除组织负责人角色）

### 1\.4 通知处理流程



业务事项完成后，系统生成对应通知并写入通知中心。学生的时效性事项同时发送移动端 Push；教师和管理员的时效性事项同时发送邮件。Push 或邮件未送达时，通知中心继续保留该事项，用户登录后可查看。



同一业务事项只生成一条通知。通知内容发生更新时，在原通知中更新处理状态和时间，不重复生成相同提醒。

## 2\. 培训与上手引导



### 2\.1 学生端引导



学生首次登录后进入三步引导：



1. 打卡：选择打卡类型，开始运动，结束运动，上传凭证并提交；

2. 成绩：查看当前学期成绩、公示状态和历史课程成绩；

3. 申请：提交免测申请和校队/社团认证申请。

    

每一步包含页面截图和简要说明。学生可跳过引导，也可在帮助中心重新查看。



### 2\.2 教师端引导



教师首次进入教师端后依次查看以下操作：



1. 核对课程和学生名单；

2. 导入学生名单；

3. 查看和处理学生打卡记录、补录学时；

4. 配置成绩项目并录入成绩；

5. 发布成绩、公示并处理申诉；

6. 清理免测、认证待办后等待管理员归档。

    

每项操作提供页面高亮说明和操作视频。引导演示不产生课程、成绩、打卡或归档数据。教师可在帮助中心重新查看任一操作。



### 2\.3 管理员手册



管理员在系统文档页面查看操作手册。手册包含学期管理、课程管理、教师和管理员账号管理、学时规则、成绩归档、系统公告、帮助中心和审计日志。



手册按系统发布版本维护，标注适用版本和更新时间。学生帮助内容、教师培训内容和管理员手册**全部提供中文和英文双语版本**。

## 4\. 客服与服务反馈



### 4\.1 支持层级



|层级|处理内容|
|---|---|
|L1 自助服务|用户通过帮助中心、常见事项和操作指引自行处理。|
|L2 业务支持|授课教师处理本课程打卡、成绩、免测、认证和课程咨询；管理员处理账号、权限、课程和系统使用事项。|
|L3 技术支持|管理员将系统故障和无法解决的事项提交给开发团队处理。|



L2 处理人员核查业务记录、回复用户和发起既有业务流程。涉及打卡时长、成绩、归档和认证结果的变更，按照对应业务流程办理并保留操作记录。



### 4\.2 反馈提交



学生和教师通过"设置—服务反馈"提交反馈。反馈内容包括：



- 服务类别；

- 反馈说明；

- 当前页面；

- 客户端版本和系统版本；

- 用户主动上传的截图。

    

截图最多上传 3 张。反馈页面不收集登录凭证、Token、Cookie、完整控制台日志和完整请求内容。



### 4\.3 工单处理流程



```Plain Text
用户提交反馈
  → 系统生成工单
  → 管理员分配处理人
  → 处理中
     ├─ 需要补充信息 → 等待用户补充 → 处理中
     ├─ 业务事项 → 转交授课教师或管理员处理
     └─ 系统故障 → 升级至开发团队
  → 处理完成
  → 用户查看处理结果
  → 工单关闭
```



用户可查看本人提交工单的状态和处理结果。处理人员记录处理说明；内部处理备注仅对处理人员和管理人员可见。



### 4\.4 服务时限



服务时限按学校工作日和工作时间计算。用户补充材料期间不计入处理时长。打卡截止日前，系统安排专项值班处理无法提交打卡等紧急事项。



|服务类别|首次响应时间|处理时间|
|---|---|---|
|无法提交打卡|4 个工作小时内|1 个工作日内|
|维护或服务恢复后的打卡补交|4 个工作小时内|1 个工作日内|

## 五、后端、API 与数据一致性



### 当前分叉



- iOS：默认 `/api/v1`

- Android：`/api`，配套 TypeScript/MySQL 后端具备较好的幂等与唯一约束

- Web：`/api`，但免测路径、上传对象、状态枚举和并发规则与 Android 后端不同

    

### 并发与重复提交



- Android 后端通过每日唯一约束和 Idempotency\-Key 提供当前最强保证

- iOS 客户端具备幂等和失败恢复，但正确性最终依赖连接的服务端

- Web 后端采用"先查累计、再插入"，缺少事务级每日唯一约束；并发请求可能同时通过

- 三端均无"一个进行中运动会话"的服务端约束

    

### 建议目标架构



应选定唯一业务后端，冻结统一 OpenAPI，并由 iOS、Android、Web 共同消费。核心写操作至少应具备：统一身份、幂等键、事务、数据库唯一约束、服务端时间、统一错误码、审计记录和跨端恢复语义。

## 17\. 数据备份与恢复



### 17\.1 备份策略



|数据类型|备份频率|保留周期|
|---|---|---|
|业务数据库|每日全量 \+ 每小时增量|滚动 30 天；每学期归档节点另做一份长期快照|
|打卡凭证（图片/视频）|实时同步副本|与凭证保留期一致|
|系统配置|每次变更前快照|系统运行期间永久，下线后 1 年|



**备份数据管理规则**:



- 备份文件全部加密存储，访问权限仅限指定运维人员，每次访问留痕

- 恢复点目标（RPO）为 1 小时：任何故障最多丢失最近 1 小时内的数据

- 每学期开学前执行一次恢复演练，验证备份可用并记录演练结果

    

### 17\.2 恢复流程



```Plain Text
发现数据异常 → 确认影响范围 → 切换系统至只读/维护模式 →
  从备份恢复数据 → 校验数据完整性 →
    恢复服务 → 通知受影响用户
```



**恢复后的用户补偿规则**: 若恢复导致最近 1 小时内提交的打卡记录丢失，系统向该时段内提交过打卡的学生发送通知，其当日提交限额重置，允许当天重新提交一次。



### 17\.3 恢复时间目标



|场景|恢复方式|恢复时间目标|

## 21\. 系统维护模式与停机通知



### 21\.1 维护模式类型



|模式|效果|适用场景|
|---|---|---|
|**正常**|全部功能可用|日常运行|
|**只读模式**|可查看，禁止写入（打卡提交、成绩修改、名单导入均拒绝）|数据迁移、备份恢复期间|
|**维护模式**|普通用户看到维护页面，无法进入系统|系统升级、重大维护|



**管理员通道**: 只读模式与维护模式下，管理员均可通过管理入口正常登录。



**维护窗口约束**: 计划内维护不安排在打卡截止日前 3 天内。



### 21\.2 通知机制



|维护类型|提前通知时间|通知方式|
|---|---|---|
|**计划内维护**|48 小时以上|系统公告 \+ 学生端推送；教师/管理员走 Web 通知中心 \+ 邮件|
|**紧急维护**|立即|开启维护模式，同时按上述渠道说明原因|
|**维护完成**|完成后|按上述渠道发送"系统已恢复"通知|



### 21\.3 维护期间数据保护



|场景|保护规则|
|---|---|
|正在进行的打卡（已点"开始运动"未结束）|计时数据在本地保留，维护结束后恢复计时，已运动时长不丢失|
|只读模式期间点击"结束运动"|计时结果在本地保留，系统恢复写入后允许补交，按开始时间所在日期计入当日限额|
|正在进行的文件上传|断点续传，恢复后从断点继续|
|未提交的草稿|本地保留，不丢失|
|正在进行的数据写入|整体回滚，恢复后提示用户重新操作|

## 23\. 异常场景与容错处理



### 23\.1 异常场景处理矩阵



|异常场景|用户看到|系统行为|
|---|---|---|
|服务暂时不可用|"服务暂时不可用，请稍后重试"（含预计恢复时间）|系统侧自动重试 3 次后转入降级|
|凭证存储不可用|"文件上传失败，请稍后重试"|移动端凭证在本地保留，恢复后学生重新提交；Web 端提示稍后重传|
|邮件发送失败|教师端展示"邮件未送达"学生列表|系统自动重试 3 次（间隔递增），仍失败的进入未送达列表，教师可一键重发|
|推送服务不可用|无感知，仅缺少推送|站内通知正常，用户下次进入系统时可见|
|网络请求超时|"请求超时，请检查网络连接后重试"|前端自动重试 1 次；打卡提交类请求重试时服务端按同一次提交处理，不产生重复记录|
|并发冲突（成绩、审核等关键数据）|"数据已被他人修改，请刷新后重试"|拒绝后写入，用户刷新后基于最新数据重新操作|
|并发修改个人资料|无提示|后提交的覆盖先提交的|



### 23\.2 打卡降级规则（v1）



打卡是系统最核心的功能。后端连续 3 次健康检查失败时进入降级模式，健康检查恢复后自动退出。



**降级期间**:



- 学生填写的打卡表单和已选凭证自动保存为本地草稿（复用草稿机制），移动端凭证原始文件在本地保留

- 界面提示"服务暂时不可用，内容已保存为草稿，恢复后请重新提交"

- 恢复后由学生手动重新提交，按提交时规则正常校验（含当日限额）

- 成绩查看展示上次同步的缓存数据，标注"数据可能不是最新"，恢复后自动刷新

    

### 23\.3 移动端离线打卡（v2 版本实现，v1 不纳入）



Android/iOS 支持无网络环境完成"开始/结束运动"核心流程：



```Plain Text
开始运动 → 本地记录开始时间（本地计时，不依赖网络）
  → 结束运动 → 检查网络
      ├── 有网络 → 正常提交
      └── 无网络 → 完整记录 + 凭证保存在本地，标记"待上传"
            → 网络恢复后按开始时间先后自动依次上传
```



**离线记录上传规则**:



- 上传时校验开始时间所在日期是否已有成功提交的记录；已有则该离线记录不入库

- 离线记录须在开始运动后 72 小时内完成同步，超时不再计入

- 服务端校验记录时间的合理性，不合理的记录拒绝并提示学生联系授课老师处理

- 离线记录在同步成功前不计入学生的累计打卡时长展示

- 离线模式仅覆盖开始/结束运动流程；成绩查看、各类申请均需联网

## 24\. 版本发布与变更管理



### 24\.1 发布分级



|类型|示例|通知方式|提前通知|
|---|---|---|---|
|**紧急修复**|安全漏洞修补|公告 \+ 通知|修复完成后通知|
|**常规更新**|Bug 修复、小优化|公告 \+ 更新日志|发布当天|
|**功能更新**|新增功能、界面改版|公告 \+ 通知 \+ 更新日志|3 天前|
|**重大变更**|业务规则变更|公告 \+ 通知 \+ 邮件 \+ 更新日志|7 天前|



**移动端紧急修复规则**: App 更新经应用商店审核存在天级延迟，紧急问题优先通过服务端开关、配置调整或公告引导在当天缓解，App 版本更新随后发布。



### 24\.2 更新日志



- 系统内置"更新日志"页面，Web 端入口在"关于"，App 端入口在"设置 → 关于 → 更新日志"

- 每次发布记录版本号、发布日期、变更内容、影响范围

    

### 24\.3 移动端版本兼容



- 服务端接口向后兼容至少 2 个大版本

- App 启动时检查服务端最低版本要求，低于要求时弹窗提示更新（含下载链接）

- 除安全漏洞修复外不强制更新

    

### 24\.4 回滚规则



- 每次发布前对数据库做快照备份

- 发布后出现严重问题时，**只回滚应用程序，不回滚数据**：用户在新版本期间提交的打卡、成绩等数据一律保留，数据问题通过修复脚本向前修正

- 仅当数据结构损坏且无法向前修复时才恢复快照；恢复前先导出快照时间点之后的用户提交数据，恢复后回放入库，无法回放的部分按补偿规则通知学生并重置当日限额

## 28\. 安全事件响应



### 28\.1 安全事件分级



|级别|定义|示例|响应时限|
|---|---|---|---|
|**P1 严重**|核心数据泄露、系统完全不可用|数据库被拖库、批量学生隐私数据泄露|30 分钟内止损|
|**P2 高危**|部分功能受影响、可疑入侵迹象|单账号被劫持、异常管理员操作|2 小时内评估|
|**P3 中危**|轻微安全隐患|登录异常增多、端口扫描|24 小时内处理|
|**P4 低危**|非紧急安全问题|安全配置优化、潜在风险|下一迭代处理|



### 28\.2 事件响应五步流程



```Plain Text
发现安全事件
    │
    ├── 第 1 步：止损（P1: 30 分钟内）
    │     ├── 暂停受影响服务
    │     ├── 禁用受影响账号
    │     ├── 必要时开启维护模式，临时关闭外部访问
    │     └── 保护日志和证据
    │
    ├── 第 2 步：评估（P1: 2 小时内）
    │     ├── 确定影响范围（哪些数据被访问/修改/导出）
    │     ├── 确定攻击路径和入口
    │     ├── 判断是否为持续性攻击
    │     └── 形成初步评估报告
    │
    ├── 第 3 步：通知（评估后 24 小时内）
    │     ├── 通知受影响用户（系统公告 + 邮件）
    │     ├── 通知学校信息安全部门
    │     ├── P1 事件上报学校主管领导
    │     ├── 涉及大量个人信息泄露时通报上级主管单位
    │     └── 对外统一口径，由指定发言人沟通
    │
    ├── 第 4 步：修复
    │     ├── 修补漏洞、恢复数据、强化防护
    │     ├── 验证修复效果
    │     └── 逐步恢复服务
    │
    └── 第 5 步：复盘
          ├── 编写事故报告（发生了什么、为什么、如何防止再发生）
          ├── 更新安全策略和流程
          └── 归档到安全知识库
```



每年组织一次安全事件桌面演练，验证流程与值班响应有效。



### 28\.3 漏洞报告机制



- 系统登录页底部提供安全联系入口

- 外部研究人员可通过 `security@bnbu.edu.cn` 报告漏洞

- 漏洞修复后经报告者同意，可在更新日志中致谢

    

### 28\.4 账号被盗处理流程



```Plain Text
学生报告账号被盗
    │
    ├── Step 1: 管理员立即重置密码
    │     - 管理员触发密码重置，原密码立即失效
    │     - 学生当前所有端的登录会话失效（需重新登录）
    │
    ├── Step 2: 验证身份
    │     - 学校邮箱验证码验证，或持学生证到体育部现场验证
    │
    ├── Step 3: 设置新密码
    │     - 管理员触发密码重置邮件 → 学生设置新密码
    │
    └── Step 4: 审查
          - 审查被盗期间的异常操作（打卡记录、个人信息修改），撤销异常操作
          - 全过程记入审计日志
```



### 28\.5 等保定级



- 系统按**等保二级**建设：处理学生个人信息与教学数据，非核心涉密数据

- 按等保二级要求落实：安全管理制度、技术防护措施、应急响应预案、日志留存 6 个月以上、每年自评一次

- 是否需要外部测评以学校所在地教育主管部门要求为准

|修改已归档成绩|教师提交修正申请 → 管理员审批 → 执行|修正记录含旧值、新值、原因、审批人|
|批量修改成绩|二次确认弹窗|弹窗展示受影响的学生数和项目|
|学期归档|另一位管理员确认（四眼原则）|确认人不可与操作人为同一人|
|学期解锁|二次确认弹窗 \+ 记录解锁原因|解锁即重新开放成绩修改，与归档同级风险|
|批量标记无效打卡|二次确认弹窗|弹窗展示受影响的学生数和记录数|
|修改学时全局规则（立即生效）|二次确认弹窗|弹窗展示变更对比（旧值 → 新值）|
|修改系统配置|二次确认弹窗|变更前自动快照，可回退上一版配置|
|删除学生账号|弹窗警告 \+ 输入学号确认|软删除，数据保留 90 天，期内可恢复|
|创建/删除管理员账号|二次确认弹窗（仅超级管理员可操作）|超级管理员不可删除自己|

|全校范围数据导出|二次确认弹窗|导出行为记入审计日志|
|数据删除申请的执行|管理员审批 → 执行|匿名化处理不可逆，执行前弹窗展示删除范围|



### 30\.3 确认弹窗规范



高风险操作确认弹窗包含以下要素：



- 警告标识 \+ 明确的操作名称

- 影响范围（具体数据量：学生数、记录数等）

- 后果说明：可恢复的操作写明恢复期限（如"30 天内可恢复"）；不可逆的操作标注"此操作不可撤销"

- 需输入确认的场景，输入内容为具体业务标识（课程代码、学号等），输入验证通过前"确认"按钮不可点击

    

### 30\.4 四眼原则（学期归档）



```Plain Text
管理员 A 发起学期归档
  → 系统检查是否存在另一位活跃管理员
    ├── 有 → 向管理员 B 展示确认请求
    │       → 管理员 B 审核归档数据 → 点击"确认归档"
    │       → 归档执行
    │       → 审计日志记录：操作人 A + 确认人 B
    └── 无 → 提示"需要至少 2 位管理员才能执行学期归档"
```



### 30\.5 留痕内容



所有高风险操作的确认行为记入审计日志：操作类型、操作人、确认人（四眼场景必填）、操作对象、影响范围、确认方式、操作时间、来源地址。

### 32\.2 下线流程



```Plain Text
Step 1: 提前通知（30 天以上）
  ├── 系统公告 + 通知全部用户，邮件通知全部教师和管理员
  └── 公告包含：下线日期、数据获取方式、替代方案

Step 2: 数据归档（下线前 7 天）
  ├── 导出全部成绩数据（Excel）
  ├── 导出全部业务数据（结构化完整备份）
  ├── 导出保留期内的打卡凭证文件打包
  ├── 导出全部审计日志
  └── 归档数据存储在校方指定介质

Step 3: 数据移交（下线前 3 天）
  ├── 移交对象：学校档案馆或 IT 部门
  ├── 移交内容：全部归档数据 + 数据结构说明文档
  ├── 移交方式：加密介质 / 学校内部存储系统
  └── 双方签署《数据移交确认书》

Step 4: 最终备份（下线当天）
  ├── 执行最终全量备份，加密存储
  └── 保留期限按学校档案管理规定执行（5 年以上）

Step 5: 生产环境数据销毁（移交确认后）
  ├── 删除生产数据库全部数据
  ├── 删除云存储中全部凭证文件
  ├── 清除生产服务器日志文件
  ├── 验证生产环境无残留数据、磁盘安全擦除
  └── 生成《数据销毁确认书》

Step 6: 服务关闭
  ├── 关闭应用服务，释放云资源
  └── 取消域名解析
```



### 32\.3 下线后数据保留

## 模块2\-用户角色、权限与认证（陈昊）



**Q1\. 体育部管理员和超级管理员的区别是什么？**



→ 超级管理员拥有全部功能权限，体育部管理员默认仅有业务管理权限（学期/课程/成绩归档/学时规则/耐力跑换算表/系统公告/业务审计日志）。超级管理员可通过界面为体育部管理员逐项开启额外权限（用户管理、系统配置、全部审计日志），但"创建/删除管理员"为超级管理员独占。



**Q2\. 到底需要几个管理员？**



→ 超级管理员 \>= 2 人（避免单人故障）；体育部管理员不做硬性限制。



**Q3\. 教师、管理员、学生的账户到底应该如何注册？是否和 AutoLab 一致发送邮件重置密码进行注册？还是说由系统统一创建账号密码，账号密码和学校账号密码同步？**



→ 沿袭之前的管理员体系，和 AutoLab 一致发送邮件重置密码进行注册。不设默认密码，每个用户通过学校邮箱独立设置密码。新生由教师导入名单时自动创建账号并发送密码设置邮件（7 天有效）；老生忘记密码通过"忘记密码"功能获取重置邮件（30 分钟有效）。

## 模块5\-成绩评定与归档（何天一）



**Q1\. "体育打卡"小时如何变成百分制分数？（请给公式或选定）**



→ A\. 线性：min\(100, 有效小时/目标小时\*100\)（目标含抵扣后）。即 `体育打卡分 = min(100, 有效学时 / 目标学时 * 100)`



**Q2\. 公示期默认 5 天：按自然日还是工作日？是否允许管理员按课程缩短？**



→ A\. 自然日固定 5 天。管理员可按课程配置为 1\~7 个自然日。



**Q3\. 成绩项目是否提供"学校默认四块模板"（打卡/专项/平时/体测）？教师能否删掉其中必选块？**



→ 有模板，可调权重。四种类型不可删除，但名称和权重可改。



**Q4\. 未录入、缺考、免测、0 分——总分怎么算？**



→ 缺考=0 且权重保留；未录入则禁止发布成绩。教师可自定义调整权重。（v3 更新：已删除缓考机制）



**Q5\. 仍有未结案成绩项时，课程能否归档？学期能否归档？**



→ （v3 更新：已删除缓考机制）教师发布成绩前确保所有成绩项非"未录入"。管理员独立发起课程归档，预检通过即可归档。



**Q6\. 补考分数是否限制最高分？**



→ （v3 更新：本系统不设缓考/补考机制，此问题不再适用）



**Q7\. "预检不通过"时能否提交归档？**



→ 管理员独立发起并完成课程归档，教师端不负责归档操作。归档预检不通过时管理员可强制归档或退回修正。



**Q8\. 学生端耐力跑换算是？**



→ A\. 纯试算，不进正式成绩。



**Q9\. 管理员启用新换算表版本后，未归档课程已录入的用时是否重算分数？**



→ C\. 教师一键确认后重算。



**Q10\. 本校是否确认使用"大一/大二 vs 大三/大四"两组，而不按精确年龄表？**



→ A\. 确认用年级四组（男\-大一大二、男\-大三大四、女\-大一大二、女\-大三大四）。换算表成绩颗粒度待确认。



**Q11\. 申诉被教师驳回后，是否允许升级给管理员？几次？**



→ A\. 不允许。教师驳回即为最终结果。



**Q12\. 教师采纳申诉改分后，公示期如何处理？**



→ A\. 不延长公示期。



**Q13\. 正式成绩导出是否必须符合教务固定列模板？模板谁维护？**



→ 必须符合教务固定列模板，仍然需要和老师沟通模板细节，方便开发。（**待进一步沟通**）



**Q14\. 体测除耐力跑外的项目（跳远、仰卧起坐等）在本模块范围？**



→ 体测外包，不需要接入。v1 仅耐力跑自动换算，其余手填分数。



**Q15\. 挂科/重修是否纳入本模块？**



→ C\. 仅挂科标记，重修走教务。
