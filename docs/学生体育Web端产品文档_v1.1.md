# BNBU 学生体育 Web 端产品文档

> 文档版本：v1.1  
> 编写日期：2026-07-12  
> 产品形态：面向学生的移动优先 Web/H5  
> 视觉基准：`BNBU-web/frontend` 现有蓝白 Material 设计  
> 交付形态：单一学生 Web 端；打卡优先，覆盖 Android 学生端主要业务

## 1. 产品概述

### 1.1 背景

BNBU 学生体育业务目前由 Android、iOS 和教师/管理 Web 三端分别承载。Android 学生端已经包含首页、课程、打卡、成绩、个人中心，以及耐力跑成绩换算和免测申请等二级能力。部分华为、鸿蒙及其他设备无法安装 Android APK，这些学生不仅无法上传运动凭证，也无法完整查看课程、学时、成绩、通知和免测申请状态。

因此，本项目定位为 Android 学生端的移动 Web 适配版。网页端在功能上复刻学生最常用的体育业务，但重新按浏览器场景简化信息层级：

1. 把“打卡”放在首要位置，登录后可直接进入打卡工作区。
2. 保留首页的学时进度、风险提示、行动入口、周计划、近期任务和通知。
3. 保留课程列表、课程详情、任务和相关记录。
4. 保留成绩预估、四项权重、计算公式、缺失项和数据来源。
5. 保留学生身份、组织抵扣、通知和设置。
6. 提供耐力跑成绩换算与 800m/1000m 免测申请及申请记录。

### 1.2 产品目标

- 让无法安装学生 App 的用户通过浏览器完成主要学生体育业务。
- 以打卡为第一入口，确保最常用任务路径最短。
- 在不照搬原生控件的前提下，覆盖 Android 学生端的核心功能范围。
- 保持与现有教师/管理 Web 一致的品牌、颜色、字体和组件语言。
- 支持真实 API 优先、演示数据兜底，确保开发联调和现场验收均可进行。
- 对华为浏览器、鸿蒙浏览器、Chrome、Edge 和 Safari 提供移动端基础兼容。

### 1.3 成功标准

学生可以在 390px 宽手机视口下完成登录、打卡、课程查看、成绩查看、通知处理、耐力跑换算和免测申请。所有页面无横向滚动、无桌面侧栏、无被遮挡的主操作按钮；打卡仍是最突出、最容易抵达的入口。

## 2. 用户与使用场景

### 2.1 核心用户

- 无法安装 Android APK 的华为、鸿蒙及其他设备学生。
- 临时无法使用学生 App，但需要完成当天打卡的学生。
- 使用普通桌面浏览器进行补充查看或演示验收的学生和项目成员。

### 2.2 核心场景

- 学生在操场完成运动后，直接用手机浏览器拍摄/选择凭证并提交。
- 学生在弱网环境下查看单个文件的上传进度和失败原因。
- 学生提交后返回记录列表，确认记录已进入“待审核”。
- 学生查看被驳回或需补材料记录的教师反馈。
- 学生查看本学期课程、课程任务和相关打卡记录。
- 学生查看成绩预估、四项成绩权重和缺失项风险。
- 学生查看校队/社团抵扣状态和未读通知。
- 学生换算 800m/1000m 耐力跑成绩。
- 学生提交免测申请、上传证明并查看审核状态。
- 验收人员在后端不可用时使用演示账号走通完整流程。

## 4. 产品信息架构

### 4.1 一级导航

移动端底部导航按以下顺序排列：

1. 打卡
2. 首页
3. 课程
4. 成绩
5. 我的

“打卡”是第一项和登录后的默认页，并使用更明确的主色图标或中间强调按钮。首页不再承担默认落地页职责，而是作为学生查看全局状态的仪表盘。

### 4.2 打卡工作区

打卡页顶部使用三个分段标签：

- 任务：可参与任务、任务筛选、任务要求与截止时间。
- 提交：自主打卡或从任务带入的打卡表单。
- 记录：历史记录、审核状态、草稿、补交材料入口。

页面初次进入默认展示“提交”，减少高频打卡操作步骤。若存在需补材料记录，在顶部显示醒目的行动提示。

### 4.3 首页

- 本学期课程相关、其他运动和总学时进度。
- 风险提示：学时缺口、待审核、临近截止、成绩缺失。
- 行动入口：继续打卡、补交材料、查看成绩、处理免测申请。
- 周计划和本周完成情况。
- 近期任务。
- 通知摘要与未读数量。

### 4.4 课程

- 课程列表使用 `courseCode / Section` 作为稳定标识。
- 课程详情展示课程教师、学期、目标学时和完成进度。
- 课程任务展示要求、开放时间、截止时间和提交状态。
- 相关记录展示与课程关联的打卡记录。

### 4.5 成绩

- 当前总分预估。
- 打卡 25%、专项考试 30%、平时表现 20%、体测 25%。
- 每一项展示当前分数、权重、加权结果和数据状态。
- 展示计算公式、四舍五入规则、缺失项和数据来源。
- 待审核打卡不计入正式预估，并给出解释。

### 4.6 我的与二级功能

- 学生身份信息。
- 校队/社团抵扣状态。
- 通知中心与已读状态。
- 耐力跑成绩换算。
- 800m/1000m 免测申请与申请记录。
- 基础设置、演示数据重置和退出登录。

## 6. 核心业务流程

### 6.1 登录

```text
输入账号和密码
  → POST /api/auth/login
  → 校验 user.role === "student"
  → 保存 token 与必要用户信息
  → 请求 /api/auth/me 验证会话
  → 进入打卡工作区的“提交”标签
```

登录失败需区分：

- 账号或密码错误。
- 登录尝试过于频繁。
- 网络不可用。
- 账号不是学生角色。
- 服务暂时不可用，可选择进入演示模式。

### 6.2 打卡提交

```text
填写运动信息
  → 选择文件
  → 前端校验类型、大小和数量
  → 生成本地预览
  → 上传有效文件
  → 获得服务端文件标识
  → 所有文件上传成功
  → 二次确认
  → POST /api/sport/records
  → 防止重复点击
  → 跳转记录详情或记录列表
```

### 6.3 记录查看

```text
GET /api/sport/records
  → 隐藏“系统抵扣”等非学生上传记录
  → 按提交时间倒序
  → 点击记录
  → GET /api/sport/records/:id
  → 展示申报、凭证、审核状态和教师反馈
```

### 6.4 课程与任务

```text
进入课程
  → 查看课程列表与完成进度
  → 打开课程详情
  → 查看课程任务和相关记录
  → 从可提交任务跳转打卡页
  → 自动带入 courseId、taskId 与任务要求
```

### 6.5 成绩查看

```text
进入成绩
  → 展示总分预估和四项权重
  → 展开计算公式
  → 标记缺失项、待审核项和风险
  → 展示各项数据来源与更新时间
```

成绩页面只读，不允许学生修改教师录入结果。

### 6.6 耐力跑成绩换算

```text
读取学生性别与年级
  → 自动选择 800m 或 1000m 规则
  → 学生输入分、秒
  → POST /api/scoring/convert-endurance
  → API 不可用时使用本地完整规则表
  → 展示分数、等级和采用规则
```

### 6.7 免测申请

```text
打开免测申请
  → 查看历史申请与状态
  → 新建申请
  → 选择 800m 或 1000m
  → 填写原因并上传证明
  → 二次确认
  → POST /api/student/exemptions
  → 返回申请记录并显示待审核
```

### 6.8 通知

```text
首页或我的显示未读数量
  → 进入通知中心
  → 按全部/未读/审核反馈筛选
  → 打开通知并标记已读
  → 可执行全部已读
```

## 7. 表单规则

### 7.1 学时

- 默认值：1 小时。
- 目标可选范围：0.5–2 小时。
- 建议步进：0.5 小时。
- 每日累计不得超过后端 `rule.dailyLimit`；当前服务端规则为 2 小时。
- 前端展示剩余可申报学时，不把上限只写死在组件中。

### 7.2 运动项目

| 显示名称 | 提交值 |
|---|---|
| 跑步 | `running` |
| 篮球 | `basketball` |
| 足球 | `football` |
| 羽毛球 | `badminton` |
| 游泳 | `swimming` |
| 健身 | `fitness` |
| 骑行 | `cycling` |
| 其他 | `other` |

- 运动项目允许为空，以兼容现有 Android 逻辑。
- 选择“其他”后，自定义项目名称必填，最多 32 个字符。
- 若当前后端未保存 `sportType`，前端适配层需在提交前与后端契约对齐，不得静默丢失用户输入。

### 7.3 补充说明

- 建议最少 5 个字符，最多 300 个字符。
- 引导文案：填写时间、地点和运动内容，例如“今天在南区操场完成 5km 慢跑”。
- 演示模式和正式模式使用相同校验。

### 7.4 凭证规则

目标产品规则：

- 最多 6 张图片。
- 最多 1 个视频。
- 图片格式：JPEG、PNG、WebP、HEIC、HEIF。
- 视频格式：MP4、MOV。
- 单张图片不超过 8 MB。
- 单个视频不超过 100 MB。
- 至少上传 1 个有效凭证才能提交。

每个本地文件状态：

```ts
type UploadStatus = "waiting" | "uploading" | "success" | "failed";
```

文件对象至少包含：

```ts
interface LocalProofFile {
  id: string;
  file: File;
  previewUrl: string;
  mediaType: "image" | "video";
  mimeType: string;
  size: number;
  status: UploadStatus;
  progress: number;
  serverKey?: string;
  url?: string;
  error?: string;
}
```

### 7.5 免测申请

- 申请项目：800m 或 1000m。
- 默认项目可根据学生性别预选，但允许学生在申请页面确认。
- 申请原因必填，建议 5–500 个字符。
- 证明文件沿用统一凭证上传组件和服务端安全校验。
- 同一项目已有“待审核”申请时不允许重复提交。
- 提交按钮在请求期间禁用，成功后返回“我的申请”。
- 申请记录展示项目、原因、证明数量、提交时间、状态、审核人和审核意见。

### 7.6 耐力跑成绩换算

- 男生默认使用 1000m，女生默认使用 800m。
- 性别或年级缺失时提示先完善个人资料。
- 分钟和秒分别输入；只接受非负整数，秒数范围为 0–59。
- 总秒数必须大于 0。
- 换算结果展示分数、优秀/良好/及格/不及格等级、输入时间和采用的人群规则。
- API 不可用时使用与服务端相同的本地 80 行规则表，结果区明确标记“本地规则换算”。

## 8. 接口与当前后端差异

### 8.1 使用接口

| 用途 | 方法与路径 |
|---|---|
| 登录 | `POST /api/auth/login` |
| 当前用户 | `GET /api/auth/me` |
| 退出 | `POST /api/auth/logout` |
| 学时汇总 | `GET /api/sport/summary` |
| 学生任务 | `GET /api/student/tasks` |
| 学生身份与抵扣 | `GET /api/sport/identity` |
| 学生资料 | `GET /api/student/profile` |
| 上传凭证 | `POST /api/upload/proof` |
| 提交打卡 | `POST /api/sport/records` |
| 记录列表 | `GET /api/sport/records` |
| 记录详情 | `GET /api/sport/records/:id` |
| 补交材料 | `POST /api/sport/records/:id/supplements` |
| 通知列表 | `GET /api/common/notifications` |
| 通知已读 | `PUT /api/common/notifications/:id/read` |
| 耐力跑换算 | `POST /api/scoring/convert-endurance` |
| 免测申请列表 | `GET /api/student/exemptions` |
| 提交免测申请 | `POST /api/student/exemptions` |

所有登录后请求使用：

```http
Authorization: Bearer <token>
```

API Base URL 默认使用同源 `/api`，正式环境由 Nginx 反向代理，避免 HTTPS 混合内容和不必要的 CORS 问题。

### 8.2 已发现的契约差异

需求附件与当前 `BNBU-web/backend/server.js` 存在以下差异，必须在真实视频上传验收前处理：

| 项目 | 目标产品规则 | 当前 BNBU-web 后端 |
|---|---|---|
| 图片数量 | 6 张 | 5 张 |
| 视频数量 | 1 个 | 不支持 |
| 图片大小 | 8 MB | 10 MB |
| 视频大小 | 100 MB | 不支持 |
| 上传响应 | 文件元数据或统一标识 | `{ urls, count }` |
| `proofFiles` | 可包含媒体元数据 | 仅接受 `/uploads/...` 图片 URL 字符串 |
| `sportType` | 需要提交和回显 | 当前记录接口未保存该字段 |
| 课程详情 | 学生课程详情、任务和相关记录 | 仅能从 `/sport/summary` 和 `/student/tasks` 组合，缺少独立详情接口 |
| 成绩 | 总分、四项明细、公式与来源 | 当前后端缺少学生成绩查询接口 |
| 通知全部已读 | 一次标记全部通知 | 当前后端只有单条已读接口 |

处理原则：

1. 前端通过 API Adapter 隔离服务端差异，不在页面组件中硬编码响应格式。
2. 演示模式按目标产品规则完整展示图片和视频流程。
3. 真实模式在后端升级完成前显示准确能力提示，不伪装视频上传成功。
4. 若本次验收要求真实视频落盘，则后端上传、字段存储、静态服务和 Nginx 配置属于 P0 联调任务。
5. 课程详情优先组合现有汇总、任务和记录数据；成绩模块在专用接口完成前使用明确标注的数据适配或演示数据，不把模拟成绩冒充正式成绩。
6. 新增学生成绩接口时应由后端返回原始四项成绩、权重、加权结果、缺失项、数据来源和更新时间，前端只负责展示与可核验计算。

### 8.3 登录请求

当前后端实际只读取 `account` 和 `password`。前端仍可发送完整客户端上下文：

```json
{
  "account": "学生账号",
  "password": "密码",
  "role": "student",
  "clientType": "web"
}
```

前端必须根据响应中的 `user.role` 再次确认学生身份。

### 8.4 自主运动打卡固定值

```json
{
  "creditType": "其他运动",
  "courseId": null,
  "taskId": null
}
```

## 9. 演示模式

演示模式不是静态截图，必须能完成完整交互：

- 使用演示学生身份进入系统。
- 修改学时、运动项目和说明。
- 选择本地图片/视频并显示预览。
- 模拟逐文件上传进度、失败和重试。
- 完成二次确认和提交。
- 新记录立即出现在列表首位。
- 点击后能查看详情和凭证。
- 首页展示学时进度、风险、周计划、近期任务和通知摘要。
- 课程列表、课程详情、任务和相关记录可浏览。
- 成绩页面展示总分、四项权重、公式、缺失项和数据来源。
- 我的页面展示身份、组织抵扣、通知和设置。
- 耐力跑成绩换算可以根据不同人群与时间返回对应分数。
- 免测申请可以创建并立即出现在申请记录中。
- 通知可以筛选、打开和标记已读。
- 刷新后保留必要演示状态；提供“重置演示数据”。

页面需明显显示“演示模式（未连接后端）”，避免用户误以为数据已正式提交。

## 10. 视觉设计规范

学生端以现有 `BNBU-web/frontend` 为唯一视觉基准。

### 10.1 颜色 Token

| Token | 值 | 用途 |
|---|---|---|
| Primary | `#165DFF` | 主按钮、选中状态、链接 |
| Primary Light | `#4080FF` | 渐变和悬停 |
| Primary Soft | `#E8F3FF` / `rgba(22,93,255,.08)` | 选中背景、提示块 |
| Page | `#F5F7FA` | 页面背景 |
| Surface | `#FFFFFF` | 卡片、输入框、导航 |
| Text | `#1D2129` | 标题和主要文字 |
| Text Body | `#4E5969` | 正文 |
| Muted | `#86909C` | 辅助信息 |
| Border | `#E5E6EB` | 分割线和控件边框 |
| Success | `#00B42A` | 已通过、上传成功 |
| Warning | `#FF7D00` | 待审核、上传中 |
| Danger | `#F53F3F` | 驳回、失败、删除 |

### 10.2 字体

```css
font-family: Inter, "PingFang SC", "Source Han Sans SC", "Microsoft YaHei", sans-serif;
```

- 页面标题：24–28px，600。
- 卡片标题：16–20px，600。
- 正文和输入：14px，400。
- 字段标签：13px，500。
- 辅助信息：12px，400。
- 核心学时数字：24–32px，700。

### 10.3 间距与尺寸

- 基础间距单位：4px。
- 常用组件间距：8px、12px、16px、24px、32px。
- 手机页面左右边距：16px。
- 桌面内容最大宽度：960px；表单主体建议 560–680px。
- 卡片内边距：16–24px。
- 输入框高度：44–48px。
- 主按钮高度：至少 44px。
- 底部导航需适配 `env(safe-area-inset-bottom)`。

### 10.4 圆角和阴影

- 大卡片圆角：12px。
- 输入框和按钮圆角：8px。
- 状态标签圆角：4px 或胶囊形。
- 默认阴影：`0 2px 8px rgba(0,0,0,.06)`。
- 悬停阴影：`0 4px 16px rgba(22,93,255,.12)`。

### 10.5 按钮

- 主按钮：蓝色或蓝色渐变、白字、整行显示。
- 次按钮：白底、浅灰边框、深灰文字。
- 危险操作：红色文字或红色弱背景，不与主按钮争夺层级。
- 禁用状态：`#F2F3F5` 背景、`#86909C` 文字。
- 上传中和提交中按钮必须显示明确文案，不能只有无限旋转图标。

## 11. 响应式布局

### 11.1 断点

- `≤ 600px`：手机单列布局。
- `601–900px`：平板单列或局部双列。
- `> 900px`：桌面居中内容区。

### 11.2 手机端

- 登录页沿用现有品牌区在上、登录卡在下的结构。
- 品牌区隐藏非必要功能卡，只保留品牌、标题、说明和年份。
- 业务页使用顶部栏 + 单列内容 + 底部导航。
- 不使用教师端桌面侧栏；现有教师端在 390px 下会形成过长导航，不适合学生打卡。
- 记录列表采用卡片，不强行压缩成桌面表格。
- 上传预览采用 2–3 列网格，视频占整行或较大卡片。
- 主操作区域不得被底部导航遮挡。

### 11.3 桌面端

- 登录页复用现有双栏结构。
- 业务页居中显示，保留足够留白。
- 打卡表单宽度不超过 680px。
- 首页可使用 2–3 列数据卡，课程与成绩页面可使用主内容 + 辅助信息的双栏结构。
- 桌面端可使用顶部主导航或紧凑侧栏，但不能改变移动端“打卡优先”的一级导航顺序。

## 12. 状态与错误处理

### 12.1 页面状态

- 首次加载。
- 正常数据。
- 空记录。
- 网络失败。
- 未登录或 Token 失效。
- 后端不可用、可进入演示模式。
- 今日已达上限。

### 12.2 上传状态

- 等待上传。
- 上传中，显示百分比。
- 上传成功，显示绿色状态。
- 上传失败，显示失败原因与“重试”。
- 文件无效，显示类型/大小/数量的具体问题。

### 12.3 防重复提交

- 上传或提交期间禁用主按钮。
- 业务提交生成客户端提交锁。
- 接口返回前不允许再次触发。
- 成功后清空当前表单并跳转结果页。

## 13. 安全与隐私

- 不把文件转为 Base64 放入 JSON。
- 不在页面代码中硬编码真实 Token 或数据库信息。
- 正式环境使用 HTTPS 和同源 `/api`。
- Token 失效时清理本地会话并返回登录页。
- 图片和视频预览 URL 在组件销毁时调用 `URL.revokeObjectURL`。
- 前端校验不能替代后端 MIME、大小和数量校验。
- 记录详情只能读取当前登录学生自己的数据。
- 页面文案提醒学生不要上传与运动无关的隐私材料。

## 14. 可访问性与易用性

- 所有表单项有可见标签。
- 错误信息与字段关联，不只用颜色表达。
- 点击区域不小于 44×44px。
- 键盘可完成登录、表单填写和提交。
- 聚焦状态清晰可见。
- 图片提供可理解的替代文本；视频提供原生控制条。
- 遵守 `prefers-reduced-motion`，减少不必要动画。

## 18. 实施约束与决策

- 技术栈优先服从当前 `BNBU-web/frontend` 的零构建静态 SPA 结构，减少部署复杂度。
- 不引入大型 UI 框架；通过共享 CSS Token 复用现有视觉语言。
- 使用单一学生端入口，按业务域拆分认证、打卡、课程、成绩、通知、免测和 API 适配模块。
- 打卡作为默认路由和一级导航第一项，但首页、课程、成绩和我的必须具备完整可验收页面。
- 页面结构按移动端重新设计，不复用教师端侧栏布局。
- 演示模式用于可验收兜底，不能掩盖真实接口能力差异。
- 在真实视频上传后端契约未升级前，不宣称正式环境已支持视频落盘。
