# BNBU 学生运动打卡 Web 端产品文档

> 文档版本：v1.0  
> 编写日期：2026-07-12  
> 产品形态：面向学生的移动优先 Web/H5  
> 视觉基准：`BNBU-web/frontend` 现有蓝白 Material 设计  
> 交付形态：两个可独立运行、业务闭环一致、信息架构不同的版本

## 1. 产品概述

### 1.1 背景

BNBU 学生体育业务目前由 Android、iOS 和教师/管理 Web 三端分别承载。部分华为、鸿蒙及其他设备无法安装 Android APK，学生因此无法完成运动凭证上传和自主运动打卡。

本项目不是把 Android 五个模块完整搬到浏览器，而是把学生打卡的最短闭环稳定迁移到移动浏览器：

1. 学生登录。
2. 填写本次运动信息。
3. 选择或拍摄图片、选择视频。
4. 查看逐文件上传进度。
5. 二次确认并提交打卡。
6. 在记录列表和详情页确认审核状态。

### 1.2 产品目标

- 让无法安装学生 App 的用户在手机浏览器中完成运动打卡。
- 保持与现有教师/管理 Web 一致的品牌、颜色、字体和组件语言。
- 同时提供两个可独立运行的学生端版本，用于比较“最短操作路径”和“状态引导路径”。
- 支持真实 API 优先、演示数据兜底，确保开发联调和现场验收均可进行。
- 对华为浏览器、鸿蒙浏览器、Chrome、Edge 和 Safari 提供移动端基础兼容。

### 1.3 成功标准

学生可以在 390px 宽手机视口下完成登录、填写、选取凭证、看到上传状态、提交打卡、查看记录和详情，整个过程中无横向滚动、无桌面侧栏、无被遮挡的主操作按钮。

## 2. 用户与使用场景

### 2.1 核心用户

- 无法安装 Android APK 的华为、鸿蒙及其他设备学生。
- 临时无法使用学生 App，但需要完成当天打卡的学生。
- 使用普通桌面浏览器进行补充查看或演示验收的学生和项目成员。

### 2.2 核心场景

- 学生在操场完成运动后，直接用手机浏览器拍摄/选择凭证并提交。
- 学生在弱网环境下查看单个文件的上传进度和失败原因。
- 学生提交后返回记录列表，确认记录已进入“待审核”。
- 学生查看被驳回或需补材料记录的教师反馈。
- 验收人员在后端不可用时使用演示账号走通完整流程。

## 3. 产品范围

### 3.1 P0：本次必须交付

- 学生登录与 Token 管理。

- 失败文件重试。

- 上传期间禁止重复提交。
- 提交前二次确认。
- 打卡记录列表、状态筛选和空状态。
- 打卡详情、凭证预览、审核结果和教师反馈。

### 3.2 P1：可在后续迭代

- 多主题和个性化设置。

## 4. 两个交付版本

两套版本复用同一套业务规则、接口适配层和视觉 Token，但入口结构、内容组织和提交节奏不同。两套均需独立通过验收，不以其中一套作为另一套的静态截图或跳转别名。

### 4.1 Version A：快速打卡版

定位：让熟悉规则的学生用最短路径完成当天打卡。

登录后默认进入“提交打卡”，页面按单列连续表单组织：

1. 今日状态提示。
2. 本次学时。
3. 运动项目。
4. 补充说明。
5. 凭证上传。
6. 提交操作。

底部导航：

- 打卡
- 记录
- 我的

特点：

- 操作路径短，适合高频用户。
- 所有必填项和上传状态在同一页面可见。
- 主按钮固定在内容末尾；移动端可增加安全区内的吸底操作栏。
- 适合快速打卡和紧急补录场景。

### 4.2 Version B：状态首页 + 分步向导版

定位：先帮助学生理解本学期进度和当前状态，再通过分步引导降低误操作。

登录后默认进入“首页”，展示：

- 本学期累计有效学时。
- 剩余学时。
- 待审核数量。
- 今日是否可提交。
- 最近一条记录。
- “开始打卡”主按钮。

提交向导分三步：

1. 运动信息：学时、运动项目、补充说明。
2. 上传凭证：选择、预览、校验、上传与重试。
3. 确认提交：汇总本次信息并二次确认。

底部导航：

- 首页
- 打卡
- 记录
- 我的

特点：

- 信息层级更清楚，适合首次使用和不熟悉规则的学生。
- 每一步只处理一类任务，错误定位更明确。
- 可在上传完成后返回确认页，不丢失已填写内容。

## 5. 页面与路由

建议两个版本放在同一前端目录下并使用独立入口：

```text
/student/a/                     Version A 入口
/student/a/login                A 登录
/student/a/check-in             A 快速打卡
/student/a/records              A 记录列表
/student/a/records/:id          A 记录详情
/student/a/profile              A 我的

/student/b/                     Version B 入口
/student/b/login                B 登录
/student/b/home                 B 状态首页
/student/b/check-in/info        B 步骤 1：运动信息
/student/b/check-in/proofs      B 步骤 2：上传凭证
/student/b/check-in/confirm     B 步骤 3：确认提交
/student/b/records              B 记录列表
/student/b/records/:id          B 记录详情
/student/b/profile              B 我的
```

若采用纯静态 SPA，可使用 hash 路由实现相同语义，避免服务器额外配置 history fallback。

## 6. 核心业务流程

### 6.1 登录

```text
输入账号和密码
  → POST /api/auth/login
  → 校验 user.role === "student"
  → 保存 token 与必要用户信息
  → 请求 /api/auth/me 验证会话
  → 进入对应版本默认页
```

登录失败需区分：

- 账号或密码错误。
- 登录尝试过于频繁。
- 网络不可用。
- 账号不是学生角色。
- 服务暂时不可用，可选择进入演示模式。

### 6.2 打卡提交

```text
填写运动信息
  → 选择文件
  → 前端校验类型、大小和数量
  → 生成本地预览
  → 上传有效文件
  → 获得服务端文件标识
  → 所有文件上传成功
  → 二次确认
  → POST /api/sport/records
  → 防止重复点击
  → 跳转记录详情或记录列表
```

### 6.3 记录查看

```text
GET /api/sport/records
  → 隐藏“系统抵扣”等非学生上传记录
  → 按提交时间倒序
  → 点击记录
  → GET /api/sport/records/:id
  → 展示申报、凭证、审核状态和教师反馈
```

## 7. 表单规则

### 7.1 学时

- 默认值：1 小时。
- 目标可选范围：0.5–2 小时。
- 建议步进：0.5 小时。
- 每日累计不得超过后端 `rule.dailyLimit`；当前服务端规则为 2 小时。
- 前端展示剩余可申报学时，不把上限只写死在组件中。

### 7.2 运动项目

| 显示名称 | 提交值 |
|---|---|
| 跑步 | `running` |
| 篮球 | `basketball` |
| 足球 | `football` |
| 羽毛球 | `badminton` |
| 游泳 | `swimming` |
| 健身 | `fitness` |
| 骑行 | `cycling` |
| 其他 | `other` |

- 运动项目允许为空，以兼容现有 Android 逻辑。
- 选择“其他”后，自定义项目名称必填，最多 32 个字符。
- 若当前后端未保存 `sportType`，前端适配层需在提交前与后端契约对齐，不得静默丢失用户输入。

### 7.3 补充说明

- 建议最少 5 个字符，最多 300 个字符。
- 引导文案：填写时间、地点和运动内容，例如“今天在南区操场完成 5km 慢跑”。
- 演示模式和正式模式使用相同校验。

### 7.4 凭证规则

目标产品规则：

- 最多 6 张图片。
- 最多 1 个视频。
- 图片格式：JPEG、PNG、WebP、HEIC、HEIF。
- 视频格式：MP4、MOV。
- 单张图片不超过 8 MB。
- 单个视频不超过 100 MB。
- 至少上传 1 个有效凭证才能提交。

每个本地文件状态：

```ts
type UploadStatus = "waiting" | "uploading" | "success" | "failed";
```

文件对象至少包含：

```ts
interface LocalProofFile {
  id: string;
  file: File;
  previewUrl: string;
  mediaType: "image" | "video";
  mimeType: string;
  size: number;
  status: UploadStatus;
  progress: number;
  serverKey?: string;
  url?: string;
  error?: string;
}
```

## 8. 接口与当前后端差异

### 8.1 使用接口

| 用途 | 方法与路径 |
|---|---|
| 登录 | `POST /api/auth/login` |
| 当前用户 | `GET /api/auth/me` |
| 退出 | `POST /api/auth/logout` |
| 学时汇总 | `GET /api/sport/summary` |
| 上传凭证 | `POST /api/upload/proof` |
| 提交打卡 | `POST /api/sport/records` |
| 记录列表 | `GET /api/sport/records` |
| 记录详情 | `GET /api/sport/records/:id` |

所有登录后请求使用：

```http
Authorization: Bearer <token>
```

API Base URL 默认使用同源 `/api`，正式环境由 Nginx 反向代理，避免 HTTPS 混合内容和不必要的 CORS 问题。

### 8.2 已发现的契约差异

需求附件与当前 `BNBU-web/backend/server.js` 存在以下差异，必须在真实视频上传验收前处理：

| 项目 | 目标产品规则 | 当前 BNBU-web 后端 |
|---|---|---|
| 图片数量 | 6 张 | 5 张 |
| 视频数量 | 1 个 | 不支持 |
| 图片大小 | 8 MB | 10 MB |
| 视频大小 | 100 MB | 不支持 |
| 上传响应 | 文件元数据或统一标识 | `{ urls, count }` |
| `proofFiles` | 可包含媒体元数据 | 仅接受 `/uploads/...` 图片 URL 字符串 |
| `sportType` | 需要提交和回显 | 当前记录接口未保存该字段 |

处理原则：

1. 前端通过 API Adapter 隔离服务端差异，不在页面组件中硬编码响应格式。
2. 演示模式按目标产品规则完整展示图片和视频流程。
3. 真实模式在后端升级完成前显示准确能力提示，不伪装视频上传成功。
4. 若本次验收要求真实视频落盘，则后端上传、字段存储、静态服务和 Nginx 配置属于 P0 联调任务。

### 8.3 登录请求

当前后端实际只读取 `account` 和 `password`。前端仍可发送完整客户端上下文：

```json
{
  "account": "学生账号",
  "password": "密码",
  "role": "student",
  "clientType": "web"
}
```

前端必须根据响应中的 `user.role` 再次确认学生身份。

### 8.4 自主运动打卡固定值

```json
{
  "creditType": "其他运动",
  "courseId": null,
  "taskId": null
}
```

## 9. 演示模式

演示模式不是静态截图，必须能完成完整交互：

- 使用演示学生身份进入系统。
- 修改学时、运动项目和说明。
- 选择本地图片/视频并显示预览。
- 模拟逐文件上传进度、失败和重试。
- 完成二次确认和提交。
- 新记录立即出现在列表首位。
- 点击后能查看详情和凭证。
- 刷新后保留必要演示状态；提供“重置演示数据”。

页面需明显显示“演示模式（未连接后端）”，避免用户误以为数据已正式提交。

## 10. 视觉设计规范

学生端以现有 `BNBU-web/frontend` 为唯一视觉基准。

### 10.1 颜色 Token

| Token | 值 | 用途 |
|---|---|---|
| Primary | `#165DFF` | 主按钮、选中状态、链接 |
| Primary Light | `#4080FF` | 渐变和悬停 |
| Primary Soft | `#E8F3FF` / `rgba(22,93,255,.08)` | 选中背景、提示块 |
| Page | `#F5F7FA` | 页面背景 |
| Surface | `#FFFFFF` | 卡片、输入框、导航 |
| Text | `#1D2129` | 标题和主要文字 |
| Text Body | `#4E5969` | 正文 |
| Muted | `#86909C` | 辅助信息 |
| Border | `#E5E6EB` | 分割线和控件边框 |
| Success | `#00B42A` | 已通过、上传成功 |
| Warning | `#FF7D00` | 待审核、上传中 |
| Danger | `#F53F3F` | 驳回、失败、删除 |

### 10.2 字体

```css
font-family: Inter, "PingFang SC", "Source Han Sans SC", "Microsoft YaHei", sans-serif;
```

- 页面标题：24–28px，600。
- 卡片标题：16–20px，600。
- 正文和输入：14px，400。
- 字段标签：13px，500。
- 辅助信息：12px，400。
- 核心学时数字：24–32px，700。

### 10.3 间距与尺寸

- 基础间距单位：4px。
- 常用组件间距：8px、12px、16px、24px、32px。
- 手机页面左右边距：16px。
- 桌面内容最大宽度：960px；表单主体建议 560–680px。
- 卡片内边距：16–24px。
- 输入框高度：44–48px。
- 主按钮高度：至少 44px。
- 底部导航需适配 `env(safe-area-inset-bottom)`。

### 10.4 圆角和阴影

- 大卡片圆角：12px。
- 输入框和按钮圆角：8px。
- 状态标签圆角：4px 或胶囊形。
- 默认阴影：`0 2px 8px rgba(0,0,0,.06)`。
- 悬停阴影：`0 4px 16px rgba(22,93,255,.12)`。

### 10.5 按钮

- 主按钮：蓝色或蓝色渐变、白字、整行显示。
- 次按钮：白底、浅灰边框、深灰文字。
- 危险操作：红色文字或红色弱背景，不与主按钮争夺层级。
- 禁用状态：`#F2F3F5` 背景、`#86909C` 文字。
- 上传中和提交中按钮必须显示明确文案，不能只有无限旋转图标。

## 11. 响应式布局

### 11.1 断点

- `≤ 600px`：手机单列布局。
- `601–900px`：平板单列或局部双列。
- `> 900px`：桌面居中内容区。

### 11.2 手机端

- 登录页沿用现有品牌区在上、登录卡在下的结构。
- 品牌区隐藏非必要功能卡，只保留品牌、标题、说明和年份。
- 业务页使用顶部栏 + 单列内容 + 底部导航。
- 不使用教师端桌面侧栏；现有教师端在 390px 下会形成过长导航，不适合学生打卡。
- 记录列表采用卡片，不强行压缩成桌面表格。
- 上传预览采用 2–3 列网格，视频占整行或较大卡片。
- 主操作区域不得被底部导航遮挡。

### 11.3 桌面端

- 登录页复用现有双栏结构。
- 业务页居中显示，保留足够留白。
- Version A 表单宽度不超过 680px。
- Version B 首页可使用 2–3 列数据卡，向导主体保持单列。

## 12. 状态与错误处理

### 12.1 页面状态

- 首次加载。
- 正常数据。
- 空记录。
- 网络失败。
- 未登录或 Token 失效。
- 后端不可用、可进入演示模式。
- 今日已达上限。

### 12.2 上传状态

- 等待上传。
- 上传中，显示百分比。
- 上传成功，显示绿色状态。
- 上传失败，显示失败原因与“重试”。
- 文件无效，显示类型/大小/数量的具体问题。

### 12.3 防重复提交

- 上传或提交期间禁用主按钮。
- 业务提交生成客户端提交锁。
- 接口返回前不允许再次触发。
- 成功后清空当前表单并跳转结果页。

## 13. 安全与隐私

- 不把文件转为 Base64 放入 JSON。
- 不在页面代码中硬编码真实 Token 或数据库信息。
- 正式环境使用 HTTPS 和同源 `/api`。
- Token 失效时清理本地会话并返回登录页。
- 图片和视频预览 URL 在组件销毁时调用 `URL.revokeObjectURL`。
- 前端校验不能替代后端 MIME、大小和数量校验。
- 记录详情只能读取当前登录学生自己的数据。
- 页面文案提醒学生不要上传与运动无关的隐私材料。

## 14. 可访问性与易用性

- 所有表单项有可见标签。
- 错误信息与字段关联，不只用颜色表达。
- 点击区域不小于 44×44px。
- 键盘可完成登录、表单填写和提交。
- 聚焦状态清晰可见。
- 图片提供可理解的替代文本；视频提供原生控制条。
- 遵守 `prefers-reduced-motion`，减少不必要动画。

## 15. 验收标准

## 18. 实施约束与决策

- 技术栈优先服从当前 `BNBU-web/frontend` 的零构建静态 SPA 结构，减少部署复杂度。
- 不引入大型 UI 框架；通过共享 CSS Token 复用现有视觉语言。
- 两个版本独立入口、共享底层业务模块，避免复制认证、上传和 API 错误处理。
- 页面结构按移动端重新设计，不复用教师端侧栏布局。
- 演示模式用于可验收兜底，不能掩盖真实接口能力差异。
- 在真实视频上传后端契约未升级前，不宣称正式环境已支持视频落盘。
