# 第一章 项目概述

## 1.1 产品名称

暂定名称：**校园生活平台 / Campus Life / Campus Hub**

最终名称可根据学校特色后续确定。

## 1.2 产品一句话

为 BNBU 打造一个立足本校的校园综合平台 App，长期包含体育打卡、运动角、讨论角、论坛、邀请老师喝咖啡、校友模块等功能。当前 MVP 阶段优先实现体育打卡与体育成绩管理模块，以"课程相关 10 小时 + 其他运动 10 小时"打卡标准和"体育打卡、专项考试、平时表现/签到、体测"四块成绩统一管理为核心，将体育老师的成绩流程化管理工作与学生的打卡提交行为统一纳入平台，最终实现老师端「四块成绩一键成表」与学生端「打卡进度透明可查」的双闭环。

## 1.3 产品定位

本产品不是单纯的"体育打卡 App"，而是一个面向本校学生、老师、社团、校队和校友的 **校园综合平台 App**。

平台长期目标是整合校园内的体育、社团、学术讨论、论坛交流、师生互动、校友资源等功能，形成一个立足本校特色的校园生活平台。

当前开发阶段重点实现核心模块：**体育打卡模块 + 体育成绩管理子系统**。

体育成绩管理子系统的定位是：

> 体育老师的成绩成表工具，而不是学生侧校园社区。

它解决的是老师在「导入学校成绩系统之前」的所有准备动作。学校官网最终导入或录入动作仍由老师完成，本系统负责把散乱数据变成干净、统一、可检查、可导出的成绩表。

### 不是

- 不是又一个泛校园 App
- 不是以论坛、社交、活动广场为核心的产品
- 不是替代学校官方成绩系统
- 第一阶段不是做复杂 AI 验真系统

### 是

- 是体育部内部流程标准化工具
- 是老师端的成绩汇总和出表工具
- 是学生打卡、体测、考试、平时表现的结构化数据入口
- 是未来校队/社团免修证明、实时打卡防作弊、运动数据可信化的基础平台

## 1.4 背景与问题

### 1.4.1 体育打卡管理现状

学校希望通过体育打卡、全人教育、体育课程、校队、社团等方式提高学生运动参与度，形成更强的校园体育氛围。当前学校已有打卡小程序，但流程仍然很重：

1. 学生在固定时间段进行运动打卡，例如周二、周四、周六
2. 每次打卡需要上传图片和视频，一次最多记录两个小时
3. 老师需要人工审核图片和视频是否有效
4. 学生可能重复使用同一张照片，老师不一定能识别
5. 体育成绩还包括专项考试、平时签到/课堂表现、体测等数据
6. 很多老师仍然用纸记录，再从后台拉 Excel，一个个核对、换算、汇总
7. 校队、社团运动证明目前很难覆盖普通成员，老师通常只能确认社长、副社长等少数人

### 1.4.2 老师痛点汇总

| 序号 | 痛点 | 当前表现 |
|------|------|----------|
| 1 | 收打卡时间不统一 | 有的老师按周收、有的期末一次性收、有的让学生私发截图、有的让学生填 Excel |
| 2 | 课程标准不统一 | 不同老师或课程的标准不统一，学生不知道自己到底还差多少小时、缺的是哪类小时 |
| 3 | 校队/社团证明难核实 | 学生可能说自己是校队成员或体育社团成员，但老师很难逐一核实 |
| 4 | 手工整理工作量大 | 老师需要手动收截图、看聊天记录、统计表格，期末从数小时到数天 |

### 1.4.3 项目关键优势

体育部门负责人已经当面确认痛点，并表示如果系统好用，可以推动使用。决策人、痛点人、推广渠道高度重合。

## 1.5 产品整体目标

### 1.5.1 长期目标

打造一个立足本校的校园综合平台，最终实现以下模块：

1. 体育打卡
2. 运动角
3. 校队模块
4. 社团模块
5. 讨论角
6. 论坛 / 畅所欲言区
7. 邀请老师喝咖啡
8. 校友模块
9. 我的 / 个人中心

### 1.5.2 第一阶段目标（体育成绩管理子系统）

让体育老师在一个完整学期或试点班级里完成以下流程：

1. 导入班级学生名单
2. 按学校规则配置体育成绩组成
3. 收集并管理体育打卡记录
4. 录入专项考试成绩
5. 记录签到、课堂表现和平时分
6. 录入体测原始成绩，并自动换算分数
7. 自动汇总每个学生的总成绩
8. 导出一张可供学校系统录入的成绩表

### 1.5.3 当前阶段目标

> 在完整校园平台 App 的框架下，优先完整实现体育打卡模块和体育成绩管理子系统。

当前阶段重点包括：

1. 学生完成体育打卡
2. 老师统一发布和审核课程相关体育任务
3. 管理员统一配置体育打卡标准
4. 系统区分"课程相关 10 小时"和"其他运动 10 小时"
5. 校队、社团通过组织认证帮助学生抵扣其他运动小时
6. 老师录入专项考试、签到、平时分、体测数据
7. 系统自动换算体测分并汇总总成绩
8. 老师导出最终成绩表
9. 其他模块只保留入口和接口

## 1.6 成功指标

MVP 成功不以 DAU 作为第一指标，而以老师侧流程效率和实际采用为核心：

- 老师期末出表时间从「数小时到数天」降低到「几分钟到几十分钟」
- 至少 1 位体育部负责人认可并推动试点
- 至少 1-2 个班级完成真实试点
- 老师能独立完成导入名单、录入成绩、导出表格
- 期末导出的表格字段能满足学校系统录入需求
- 老师只需要处理异常打卡，不再全量逐条审图

## 1.7 分阶段路线图

### Phase 0：需求确认与样表收集

目标：把产品边界锁死，防止做错表格。

需要拿到：

- 学校最终成绩录入系统需要的字段样式
- 现有 Excel 导出样表
- 体测换算规则表
- 体育成绩四块组成及权重标准
- 1-2 位试点老师的真实班级流程
- Boss 对 MVP 的最低验收标准

### Phase 1：MVP 成表工具 + 体育打卡

目标：用最少功能跑通老师端「四块成绩一键成表」+ 学生端「打卡进度透明可查」。

包含：

- 班级/学生导入
- 四块成绩录入
- 打卡累计
- 哈希查重
- 体测换算
- 成绩汇总
- Excel/CSV 导出
- 学生提交课程相关打卡和其他运动打卡
- 老师发布课程任务并审核
- 校队/社团名单导入与自动抵扣
- 其他校园模块入口空壳

不包含：

- 完整 GPS 防作弊
- 校队/社团完整考勤系统
- 校园社区
- 自动写入学校官网

### Phase 2：异常审核与打卡可信化

目标：进一步减少老师审核量。

包含：

- 老师只看异常
- 现场拍摄
- 时间戳校验
- GPS 场馆辅助核验
- 更清晰的学生进度反馈

### Phase 3：校队 / 社团免修证明

目标：解决普通成员身份无法证明的问题。

包含：

- 校队/社团花名册
- 负责人点名考勤
- 成员参与记录
- 与免修 10 小时规则联动
- 老师端自动认可可信记录

### Phase 4：从体育场景扩展到校园 App

只有在体育成绩工具被真实采用后，再考虑向更大的校园产品扩展，例如运动角、社团活动、讨论角等。扩展顺序应由真实活跃数据决定，而不是一开始堆模块。

# 第二章 系统整体架构

## 2.1 架构概览

系统采用前后端分离架构，以校园综合平台为主基座，体育成绩管理作为内嵌子业务模块运行。

### 端侧划分

| 端 | 载体 | 用户 | 说明 |
|----|------|------|------|
| 学生端 | 原生 App（iOS + Android） | 学生 | 查看体育打卡进度、提交打卡、查看成绩进度、查看异常/未通过记录反馈、访问其他校园模块入口 |
| 老师端 | Web | 体育任课老师 | 管理课程学生、发布课程任务、审核打卡、录入专项考试/签到/平时分/体测数据、导出成绩表 |
| 管理端 | Web | 体育部管理员/Boss | 学期管理、课程管理、老师管理、成绩规则配置、体测换算表配置、导出模板配置、全校数据看板 |
| 校队/社团端 | Web（与老师/管理端共用 Web 框架，按角色展示功能） | 校队负责人、社团负责人 | 成员名单管理、身份确认、未来扩展活动发布与考勤 |

### 客户端形态决策（v1.1 确定）

| 端 | 形态 | 决策理由 |
|----|------|----------|
| 学生端 | 原生 App | 打卡核心动作是拍照/录视频 + 上传，原生相机调用与大文件压缩上传体验最好；推送通知（审核结果、截止提醒）最可靠；作为校园综合平台的长期载体，不受小程序平台规则与能力限制 |
| 老师端 / 管理端 | Web | 录成绩、批量审核、导名单、导出 Excel 属于键盘 + 大屏场景；浏览器直接访问，无上架与发版摩擦 |
| 校队/社团负责人端 | Web | 成员名单管理属管理类操作，与老师/管理端共用 Web 工程，按角色路由，降低开发成本 |

补充约定：

1. 学生端 App 使用跨平台框架（Flutter / React Native）开发，一套代码同时产出 iOS 和 Android，不允许双端各写一套
2. 保留一个最简 H5 只读页面作为兜底：未安装 App 的学生可通过链接查看打卡进度与截止提醒（不支持提交打卡）
3. 后端 API 保持端无关设计（见第八章），未来如需扩展小程序端无需改动接口

# 第三章 用户角色与权限体系

## 3.1 角色总览

| 角色 | 编码 | 说明 |
|------|------|------|
| 学生 | student | App 主要使用者，打卡数据提交者 |
| 体育任课老师 | teacher | 课程管理者，成绩录入者 |
| 体育部管理员 | admin | 全校标准配置与基础数据管理 |
| 校队负责人 | team_manager | 校队成员身份管理 |
| 社团负责人 | club_manager | 社团成员身份管理 |
| 超级管理员 | super_admin | 系统全局管理 |

## 3.2 权限矩阵

### 3.2.1 学生权限

| 模块 | 权限 |
|------|------|
| 我的体育课程 | 查看自己的课程和进度 |
| 体育打卡进度 | 查看总进度、课程相关进度、其他运动进度、还差哪类小时 |
| 提交打卡 | 提交课程相关打卡、其他运动打卡 |
| 打卡记录 | 查看所有记录、按类型/状态筛选 |
| 运动身份 | 查看校队/社团认证状态、抵扣生效情况 |
| 成绩进度 | 查看自己四块成绩完成情况 |
| 异常记录 | 查看自己的异常/未通过记录反馈 |
| 修改限制 | 不能修改老师录入的成绩 |

### 3.2.2 体育任课老师权限

| 模块 | 权限 |
|------|------|
| 我的课程 | 查看管理的课程列表 |
| 学生名单 | 管理自己课程的学生、导入学生名单 |
| 课程任务 | 创建、编辑、删除课程相关体育任务 |
| 打卡审核 | 查看并处理本课程打卡，审核学生提交，通过/驳回/要求补充材料/修改有效小时数/添加审核备注 |
| 异常打卡 | 查看异常打卡记录并处理 |
| 专项考试 | 录入专项考试成绩，支持批量录入和批量修改，保存草稿 |
| 签到/平时分 | 记录签到、课堂表现分、缺勤/迟到/请假状态，自动汇总平时成绩 |
| 体测数据 | 录入体测原始数据，查看换算结果 |
| 成绩汇总 | 查看每位学生的分项成绩、总分、缺失项 |
| 成绩导出 | 导出本课程成绩表（Excel/CSV） |
| 数据统计 | 查看全班完成率、未完成学生名单、待审核记录、被驳回记录 |

### 3.2.3 体育部管理员权限

| 模块 | 权限 |
|------|------|

| 模块 | 权限 |
|------|------|

| 数据查看 | 只能管理自己组织的成员和考勤，不能查看学生完整体育成绩 |
| 未来扩展 | 发布训练、比赛、招新等内容 |

### 3.2.5 社团负责人权限

| 模块 | 权限 |
|------|------|

## 3.3 权限设计原则

1. **最小权限原则**：每个角色只拥有完成其核心任务所必需的最小权限集合
2. **数据隔离原则**：老师只能看自己课程的学生数据；校队/社团负责人只能看自己组织成员数据
3. **隐私保护原则**：学生姓名、学号、成绩、运动记录属于敏感数据，按角色授权、限制导出权限
4. **不碰学校官网权限**：学校官网的最终导入或录入动作由老师完成，本产品只做导入前的数据整理和导出

# 第五章 综合平台自有业务模块

## 5.1 体育打卡模块（当前完整实现）

### 5.1.1 体育打卡标准

每个学生每学期需要完成 **20 小时体育学时**，分为两类：

| 类型 | 名称 | 要求小时 | 说明 |
|------|------|----------|------|
| A 类 | 课程相关体育学时 | 10 小时 | 与课程、全人教育、老师任务相关 |
| B 类 | 其他运动学时 | 10 小时 | 自主运动、校队、社团、运动活动等 |
| 总计 | | 20 小时 | A 类和 B 类都达标才算完成 |

系统判断完成的标准不是单纯总小时数达到 20，而是：

> 课程相关 ≥ 10 小时，并且其他运动 ≥ 10 小时，才算完成体育打卡。

### 5.1.2 A 类：课程相关体育学时

**定义：** 与课程、全人教育或老师布置任务相关的体育活动。例如：

1. 体育课程课外练习
2. 全人教育体育任务
3. 老师发布的跑步任务
4. 老师指定的运动训练
5. 课程要求的运动记录

**规则：**

1. A 类必须完成 10 小时
2. A 类不能被校队或社团完全替代
3. A 类由老师发布任务，学生按任务提交证明
4. A 类提交后进入对应老师审核

### 5.1.3 B 类：其他运动学时

**定义：** 课程之外的体育运动参与。例如：

1. 自主跑步
2. 篮球
3. 足球
4. 羽毛球
5. 游泳
6. 健身
7. 校队训练
8. 体育社团活动
9. 运动角活动

**规则：**

1. B 类必须完成 10 小时
2. 普通学生可以通过自主运动打卡完成
3. 校队学生可以通过校队认证自动完成 B 类 10 小时
4. 体育社团学生可以通过社团认证自动完成 B 类 10 小时
5. 校队和社团抵扣不叠加，B 类最多计 10 小时

### 5.1.4 证明体系

为降低老师审核压力，系统将体育证明分为三个等级：

| 证明等级 | 名称 | 可信度 | 来源 | 处理方式 |
|----------|------|--------|------|----------|
| 一级 | 组织认证证明 | 最高 | 校队官方名单、体育社团名单 | 系统自动抵扣，无需学生重复上传 |
| 二级 | 活动签到证明 | 中等偏高 | 校队训练签到、社团活动签到、运动角活动签到 | 当前 MVP 预留，未来支持二维码/限时/地点签到 |
| 三级 | 学生自提交证明 | 最低 | 运动 App 截图、手环记录、场地照片、文字说明 | 老师人工审核，系统基础格式检查 |

## 5.2 校队与社团抵扣规则

### 5.2.1 设计原则

校队和社团不是体育打卡里的小功能，而是校园平台中的独立组织模块。当前阶段，它们主要通过身份认证影响体育打卡。未来阶段，它们会进入运动角，拥有活动发布、报名、签到、招新等独立功能。

### 5.2.2 校队抵扣规则

若学生是认证校队成员：

| 项目 | 规则 |
|------|------|
| 课程相关学时 | 仍需完成 10 小时 |
| 其他运动学时 | 自动完成 10 小时 |
| 总要求 | 仍为 20 小时 |
| 实际还需提交 | 课程相关 10 小时 |

学生端显示：

> 校队身份已认证，系统已为你完成其他运动 10 小时。你仍需完成课程相关体育学时 10 小时。

### 5.2.3 体育社团抵扣规则

若学生是认证体育类社团成员：

| 项目 | 规则 |
|------|------|
| 课程相关学时 | 仍需完成 10 小时 |
| 其他运动学时 | 自动完成 10 小时 |
| 总要求 | 仍为 20 小时 |
| 实际还需提交 | 课程相关 10 小时 |

学生端显示：

> 体育社团身份已认证，系统已为你完成其他运动 10 小时。你仍需完成课程相关体育学时 10 小时。

### 5.2.4 非体育类社团规则

1. 不自动抵扣体育学时
2. 如果该社团组织体育活动，活动通过审核后可以单独计入 B 类其他运动学时
3. 非体育类社团成员身份本身不产生自动减免

### 5.2.5 多身份规则

如果学生同时属于校队和体育社团，或者属于多个体育社团：

- **不叠加**。B 类其他运动要求只有 10 小时，最多只抵扣 10 小时

学生端显示：

> 你拥有多个运动身份，但其他运动学时上限为 10 小时，系统已自动计入最高可用抵扣。

# 第六章 体育成绩管理子系统完整业务

> **模块说明：** 本章内容为体育成绩管理子系统（校园综合平台的体育成绩管理子模块），涵盖班级与学生管理、成绩规则配置、体育打卡记录、专项考试成绩录入、平时表现/签到、体测成绩换算、成绩汇总与导出等完整业务。本章内容与第五章体育打卡模块存在功能衔接：第五章的打卡记录作为本章成绩汇总的组成部分。

## 6.1 班级与学生管理

### 6.1.1 功能清单

- 支持创建学期、课程、班级
- 支持导入学生名单，字段至少包括姓名、学号、班级、课程
- 支持老师查看某门课下的学生列表
- 支持按学生查看四块成绩完成情况

### 6.1.2 学期初始化流程

1. 管理员创建学期
2. 创建体育课程和班级
3. 导入学生名单
4. 配置成绩组成和权重
5. 录入体测换算规则
6. 老师确认课程信息无误

输出：一个可用于本学期的体育成绩工作台。

### 6.1.3 验收标准

- 老师能在 10 分钟内完成一个班级名单导入

## 6.2 成绩规则配置

### 6.2.1 四块成绩组成

| 序号 | 成绩组成 | 说明 |
|------|----------|------|
| 1 | 体育打卡 | 学生打卡累计小时（课程相关 10h + 其他运动 10h） |
| 2 | 专项考试 | 老师设定的专项技能考试 |
| 3 | 平时表现/签到 | 出勤、课堂表现 |
| 4 | 体测 | 国家体测标准项目 |

### 6.2.2 配置能力

- 支持配置四个成绩组成部分
- 支持按学校现有规则设置权重或分值上限
- 第一阶段不重新设计学校评分标准，只把已有标准流程化

## 6.3 体育打卡记录管理

### 6.3.1 打卡记录字段

每条记录包含：

| 字段 | 说明 |
|------|------|
| 学生 | 学号、姓名 |
| 时间 | 打卡提交时间 |
| 运动类型 | 跑步、球类等 |
| 运动类别 | 课程相关 / 其他运动 |
| 时长 | 本次打卡时长 |
| 图片/视频凭证 | 上传的证明文件 |
| 文件哈希 | 用于查重 |

### 6.3.2 打卡规则

- 支持课程相关 10 小时和其他运动 10 小时分别累计
- 支持老师查看每个学生打卡完成进度
- 支持图片哈希查重，识别同图重复上传
- 支持异常标记：重复图片、缺少凭证、时长超限

### 6.3.3 学生日常打卡流程

1. 学生进入自己的课程页面
2. 查看课程相关运动和其他运动的剩余小时数
3. 提交打卡记录（含选择打卡类型：课程相关/其他运动）
4. 系统记录时间、类型、时长、凭证
5. 系统自动检查是否重复图片、超出规则、字段缺失
6. 正常记录进入累计，异常记录进入老师待处理列表

输出：结构化的打卡数据，而不是期末才整理的图片和视频堆。

## 6.4 专项考试成绩录入

### 6.4.1 功能清单

- 老师可为课程创建专项考试项目（例如足球考试、篮球考试、跑步测试等）
- 支持按学生录入原始成绩或分数
- 支持批量录入和批量修改
- 支持保存草稿，避免录到一半丢失

### 6.4.2 老师成绩录入流程

1. 老师进入课程工作台
2. 查看学生名单和四块成绩状态
3. 录入专项考试成绩
4. 记录平时表现和签到
5. 录入体测原始成绩
6. 系统自动换算体测分并汇总
7. 老师处理少量异常打卡

输出：每个学生的分项成绩和总成绩持续更新。

## 6.5 平时表现 / 签到

### 6.5.1 功能清单

- 支持按课次记录签到
- 支持记录课堂表现分或平时分
- 支持缺勤、迟到、请假等状态
- 支持自动汇总平时成绩

### 6.5.2 签到状态

| 状态 | 说明 |
|------|------|
| 出勤 | 正常出席 |
| 缺勤 | 未出席 |
| 请假 | 已请假 |
| 迟到 | 迟到到场 |

## 6.6 体测成绩换算

### 6.6.1 功能清单

- 支持录入体测原始数据（跑步时间、跳远距离、仰卧起坐次数等）
- 支持根据学校已有国家体测标准换算表自动计算分数
- 支持老师查看原始数据和换算结果
- 支持在换算规则有调整时更新规则表

### 6.6.2 体测项目

| 项目 | 原始数据类型 | 说明 |
|------|-------------|------|
| 跑步 | 时间（分:秒） | 男生 1000m / 女生 800m |
| 跳远 | 距离（米） | 立定跳远 |
| 仰卧起坐 | 次数 | 一分钟仰卧起坐 |
| 引体向上 | 次数 | 男生项目 |
| 坐位体前屈 | 距离（厘米） | 柔韧性测试 |
| 50 米跑 | 时间（秒） | 短跑 |
| 身高体重 | 数值 | BMI 计算 |

## 6.7 成绩汇总与导出

### 6.7.1 成绩汇总

- 系统自动汇总四块成绩
- 支持查看每位学生的分项成绩、总分、缺失项
- 支持导出前异常检查（缺少体测、打卡未满、重复凭证未处理）

### 6.7.2 成绩导出

- 支持导出 Excel 或 CSV
- 导出表格字段以学校最终录入系统需要的模板为准
- 支持导出前异常检查

### 6.7.3 期末出表流程

1. 老师进入期末成绩汇总页
2. 系统提示缺失项和异常项
3. 老师处理异常或确认忽略
4. 系统生成最终成绩表
5. 老师导出 Excel/CSV
6. 老师将表格数据用于学校系统录入

输出：一张干净、可追溯、可录入学校系统的成绩表。

## 10.4 安全运维

### 10.4.1 安全措施

- 全站 HTTPS，禁止 HTTP 明文传输
- API 接口统一鉴权（JWT Token）
- 敏感数据加密存储（密码使用 bcrypt/sha256 哈希）
- 文件上传类型和大小校验
- SQL 注入防护（参数化查询）
- XSS 防护（输入输出转义）
- CORS 白名单配置
- 接口限流（防止恶意请求）
- 登录失败锁定机制
- Web 前端启用 CSP、拒绝 iframe 嵌套、禁止 MIME sniffing、referrer 策略和权限策略
- API Base URL 在前端做协议和非法字符校验；正式环境仅允许 HTTPS
- CSV 导入需在服务端进行最终校验，防止越权导入、重复导入和公式注入
- 所有成绩保存、异常审核、组织抵扣、导出和归档操作必须写入审计日志

### 10.4.2 稳定性与并发基线

- Web 前端以静态资源方式交付，可通过 CDN/Nginx/对象存储承接大量读取请求
- 当前 Web 产品联调目标为：200 名老师/管理员/组织负责人同时登录并进行常用操作
- 后端压测验收建议：200 并发用户下 P95 API 响应时间不高于 800ms，错误率低于 1%
- 写接口必须具备幂等控制，避免重复点击导致重复加学时、重复导入或重复归档
- 导出成绩、批量导入和批量换算建议使用异步任务或流式下载，避免长任务阻塞主请求线程
- API 请求前端默认 5 秒超时、最多重试 2 次；后端应返回稳定的错误码和可读错误摘要
- 网关或 API 层必须配置基础限流，异常 IP、异常账号或高频导出请求需要返回 429 并记录日志
- Web 端必须具备运行期错误兜底，单页数据异常时不能整页白屏，需提供恢复或重置入口
- 交付包需提供可复现质量烟测脚本，用于验证安全响应头、路径防护、CORS 与 200 并发基础能力

### 10.4.3 浏览器兼容基线

- 老师端、管理端和组织负责人端目标浏览器为 Chrome、Edge、Safari、Firefox 最近两个稳定版本
- 桌面端重点验证 1366px、1440px 和更大屏幕；窄屏至少保证 320px 基础降级可用
- CSV 下载需在 Chrome、Safari、Edge 中验证中文不乱码，并保留 UTF-8 BOM
- `localStorage` 不可用时前端应能回退默认状态；正式接后端后本地只保留少量 UI 偏好
- `AbortController` 不可用时允许降级，但页面必须保留错误提示和重试能力

### 10.4.4 隐私与数据权限

- 学生姓名、学号、成绩、运动记录属于敏感数据
- 最小化收集原则
- 按角色授权、限制导出权限
- 明确数据保留周期
- 打卡凭证压缩策略：可设置保留期限、只保存关键凭证

### 10.4.5 应急响应

- 制定数据泄露应急响应流程
- 建立安全漏洞报告机制
- 定期安全扫描和依赖更新

## 10.6 风险与应对

### 10.6.1 表格模板不匹配

**风险：** 导出的成绩表不符合学校系统录入习惯，老师仍然要手动整理。

**应对：** Phase 0 必须拿到真实模板，用模板倒推字段。

### 10.6.2 老师不愿换流程

**风险：** 现有小程序虽然难用，但老师已经习惯。

**应对：** 先让试点老师在期末看到「几天变几分钟」的价值，再扩大推广。

### 10.6.3 学生端使用麻烦

**风险：** 学生不愿多下一个 App 或多走一个流程。

**应对：**

1. 第一阶段学生端只保留必要动作，重点服务老师端闭环
2. 利用封闭校园分发渠道：开学第一节体育课由老师组织扫码安装
3. 用抵扣机制制造即时正反馈：校队/体育社团学生安装后立即看到「其他运动 10 小时已完成」
4. 提供 H5 只读兜底页，未安装 App 的学生也能查看进度与收到截止提醒触达

### 10.6.4 防作弊预期过高

**风险：** 承诺 AI 防作弊，但实际判断不了学生是否真运动。

**应对：** 第一阶段只承诺可确定的哈希查重和异常提示。第二阶段再做现场拍摄、GPS、场馆码。

### 10.6.5 视频和图片存储成本

**风险：** 打卡凭证占用大量存储。

**应对：** MVP 先确认是否必须保存视频原件；可设置压缩、保留期限、只保存关键凭证。

### 10.6.6 隐私与数据权限

**风险：** 学生姓名、学号、成绩、运动记录属于敏感数据。

**应对：** 最小化收集、按角色授权、限制导出权限、明确数据保留周期。

### 10.6.7 App 分发与上架合规

**风险：** 国内 iOS/安卓应用商店上架需要 ICP 备案、软件著作权等合规材料，周期长且存在不确定性，可能阻塞试点。

**应对：** 试点期采用 TestFlight（iOS）+ APK 扫码直装（Android）的轻量分发路线，完全绕开应用商店；正式上架推迟到试点验证成功之后（见 9.4 客户端分发策略）。

# 附录 A 关键产品原则

1. **先让老师少干活** — 第一阶段所有设计都围绕老师端效率。学生端不能做得太复杂，否则会增加推广阻力
2. **数据从一开始就结构化** — 不要把 Excel、PDF、截图当作最终形态。只要数据从源头结构化，期末整理工作就会自然消失
3. **不承诺 AI 魔法** — AI 可以作为未来辅助，但第一阶段不要承诺「AI 判断照片视频真假」。同图复用用哈希查重即可，真正的防作弊要靠流程设计
4. **老师只处理异常** — 系统价值不是让老师用更好的界面审更多材料，而是让老师不用审正常材料，只看异常
5. **不碰学校官网权限** — 学校官网的最终导入或录入动作由老师完成。本产品只做导入前的数据整理和导出，避免越权和集成风险

## C.3 统一 UI 风格标准（以 UI_Style.png 为准）

后续产品原型、Web 端和学生端 App 的视觉风格统一以项目目录下 `UI_Style.png` 为标准，不再采用橙色运动活力风、Bento 装饰卡片风或校园卡通风作为主方向。

- **核心风格：** Minimalism & Swiss Style，强调理性、秩序、克制和工具感
- **主视觉语言：** 大面积留白、严格网格、黑白高对比、明确分栏和清晰信息密度
- **色彩原则：** 以白色/近白背景、黑色正文和深灰辅助信息为主；强调色只用于关键状态、风险提示或主要操作，不做大面积装饰
- **排版原则：** 标题使用强对比大字号，正文保持清晰可读，数据与表格保持紧凑、整齐、可扫描
- **组件原则：** 按钮、表格、表单、筛选器和卡片边界保持克制，避免圆角过大、阴影过重、渐变背景和复杂插画
- **产品气质：** 老师端与管理端应像高效率成绩工作台，学生端应像清晰可靠的校园工具，而不是营销页或泛社交社区

# 附录 D 产品核心价值总结

本产品的核心价值不是"让学生上传截图"，而是：

1. 建立一个本校统一的体育学时标准
2. 把体育打卡从老师手动收材料变成平台统一管理
3. 把 20 小时拆成课程相关 10 小时和其他运动 10 小时，双维度进度透明
4. 让学生清楚知道自己到底还差哪一类小时
5. 让校队和体育社团通过组织认证完成可信证明
6. 将体育打卡、专项考试、平时表现、体测四类数据结构化汇总
7. 支持自动换算、异常检查和成绩表导出
8. 降低老师审核和统计成本，期末出表从数天降至数十分钟
9. 为后续运动角、讨论角、论坛、老师 coffee chat、校友模块打下平台基础

# 附录 E 最终产品表述

我们要做的是一个本校校园综合平台，而不是单独的体育打卡工具。客户端形态为：学生端为原生 App（iOS + Android，跨平台框架开发），老师端与管理端为 Web。平台长期将包含体育打卡、运动角、讨论角、论坛、邀请老师喝咖啡、校友模块等校园功能。当前 MVP 阶段优先实现体育打卡模块和体育成绩管理子系统，通过统一的"课程相关 10 小时 + 其他运动 10 小时"打卡标准和"体育打卡、专项考试、平时表现/签到、体测"四块成绩统一管理，解决老师收打卡时间不统一、课程标准不统一、校队/社团证明难核实、成绩汇总繁琐的问题。同时，校队和社团当前作为体育打卡的证明与抵扣来源，未来会扩展到运动角中，承担活动发布、训练通知、报名签到和校园体育氛围建设等功能。

体育成绩管理子系统作为内嵌业务模块，服务体育部真实流程，将体育打卡、专项考试、平时表现、体测四类数据结构化汇总，支持自动换算、异常检查和成绩表导出，推动从纸质/Excel 手工核对向标准化流程迁移。
